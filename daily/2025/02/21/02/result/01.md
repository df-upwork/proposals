1) Причина вашей проблемы: в Tenant 1 (где создаётся встреча в Teams) нет нужного механизма или не настроен процесс для автоматического создания (или синхронизации) учётных записей пользователей из Tenant 2. 
2) Когда организатор из Tenant 1 пытается назначить приглашённого пользователя из Tenant 2 сопроводителем (co-organizer), в Azure AD первого арендатора не существует гостевой записи для этого пользователя. 
Из‐за этого Microsoft Teams не может присвоить ему права, необходимые для роли организатора, ведь Teams под капотом полагается на Azure AD (или на сопоставление в локальную для арендатора директорию).
3) При этом междоменный чат (внешний федеративный чат) может работать, потому что для текстовой коммуникации требуется лишь включённая федерация между доменами/арендаторами (внешние чаты не создают полноценной учётной записи в чужом Azure AD). 
Но для роли co-organizer нужно именно существование пользователя (в виде guest или другого типа учётной записи) в Tenant 1. 
4) Для решения проблемы нужно:
4.1) Проверить настройки федерации в Microsoft Teams  
4.1.1) Открыть «Microsoft Teams admin center» в Tenant 1.  
4.1.2) Перейти в «Org-wide settings» → «External access».  
4.1.3) Убедиться, что в списке «Allowed domains» присутствует домен Tenant 2 (ugly.com) и что включена возможность внешнего взаимодействия.  
4.1.4) Выполнить то же самое в Tenant 2, проверив список «Allowed domains» для домена Tenant 1 (acme.com).  
4.2) Настроить Azure B2B Collaboration для Tenant 2 в Tenant 1  
4.2.1) Открыть «Azure Active Directory admin center» в Tenant 1.  
4.2.2) Перейти в «External Identities» → «External collaboration settings».  
4.2.3) Включить «Guest access» и разрешить отправку приглашений пользователям внешних организаций (можно ограничить по доменам, добавив в список разрешённых домен Tenant 2).  
4.2.4) При необходимости отредактировать «Collaboration restrictions», чтобы домен Tenant 2 (ugly.com) был в списке разрешённых.  
4.3) Подключить Azure AD Cross-Tenant Synchronization (по желанию, если требуется автоматическая синхронизация)  
4.3.1) В «Azure Active Directory admin center» в Tenant 1 перейти в «External Identities» → «Cross-tenant synchronization (Preview)» (если доступно в подписке).  
4.3.2) Создать новую связь синхронизации с Tenant 2, заполнив поля для идентификаторов арендатора и настроив параметры, позволяющие автоматически создавать учётные записи пользователей Tenant 2 в Tenant 1.  
4.3.3) При необходимости указать группы или все учётные записи, которые нужно синхронизировать.  
4.3.4) Сохранить настройки и дождаться завершения первоначальной синхронизации.  
4.4) Пригласить пользователей из Tenant 2 в Tenant 1 вручную (альтернативный вариант без Cross-Tenant Synchronization)  
4.4.1) В «Azure Active Directory admin center» в Tenant 1 перейти в «Users» → «New guest user».  
4.4.2) Добавить адреса электронной почты пользователей из Tenant 2 (ugly.com) в формате user@ugly.com.  
4.4.3) Отправить им приглашение. После принятия приглашения в Azure Active Directory Tenant 1 появится учётная запись гостя.  
4.4.4) Проверить, что пользователи корректно отображаются в списке «Users» → «All users» и имеют статус «Guest».  
4.5) Проверить настройки роли co-organizer в Microsoft Teams  
4.5.1) В «Microsoft Teams admin center» Tenant 1 перейти в «Meetings» → «Meeting settings».  
4.5.2) Убедиться, что в разделе «Roles» опция приглашения пользователей с ролью co-organizer не ограничена только внутренними пользователями.  
4.5.3) При создании новой встречи в Microsoft Teams (в Microsoft 365) проверить, что пользователь из Tenant 2 (добавленный как Guest или синхронизированный) доступен для выбора в списке и может быть назначен co-organizer.  
4.6) Убедиться, что почтовые сервисы Google в Tenant 1 продолжают работать без изменений  
4.6.1) Проверить MX-записи и другие записи DNS для домена acme.com, чтобы в них по-прежнему указывалась платформа Google Workspace.  
4.6.2) Убедиться, что почтовые ящики сотрудников в Tenant 1 (acme.com), использующие Google Workspace, продолжают функционировать согласно стандартным процессам Google.  
4.6.3) При необходимости отредактировать SPF/DMARC/DKIM-записи так, чтобы они не нарушали работу почты в Google Workspace и корректно учитывали приглашения, отправляемые из Microsoft Teams.  
4.7) Протестировать назначение co-organizer при планировании встречи  
4.7.1) Войти в Microsoft Teams в Tenant 1 под учётной записью организатора встречи.  
4.7.2) Создать новую встречу в «Outlook on the web» или в «Microsoft Teams» (календарь).  
4.7.3) Добавить пользователя из Tenant 2 (ugly.com) в качестве гостя (guest) или, если настроена синхронизация, выбрать его учётную запись напрямую в списке.  
4.7.4) Назначить этого пользователя co-organizer, отправить приглашение и убедиться, что со стороны Tenant 2 приглашённый пользователь получил доступ к встрече с правами co-organizer.

## **Раздел 1: Деконструкция основной проблемы (P1): Архитектура включения ODBC**

### **Цель**

Цель данного раздела — установить, что проблема P1 является не ошибкой, а особенностью архитектуры FileMaker, ориентированной на безопасность и инкапсуляцию на уровне каждого файла. Понимание этого является ключевым для оценки жизнеспособности любого решения.

### **1.1. Модель безопасности FileMaker: Триада контроля**

Платформа FileMaker использует многоуровневую модель безопасности, состоящую из трех основных компонентов, которые совместно определяют права доступа к данным и функциональности.1

* **Учетные записи:** Фундаментальная единица идентификации пользователя. Каждая учетная запись (сочетание имени пользователя и пароля) аутентифицирует пользователя при доступе к файлу.  
* **Наборы привилегий:** Это коллекции разрешений, назначаемые учетной записи. Набор привилегий детально определяет, к каким таблицам, полям, макетам, скриптам и пунктам меню пользователь имеет доступ, а также какие операции (создание, редактирование, удаление) он может выполнять.  
* **Расширенные привилегии:** Это особый тип разрешений внутри набора привилегий, который регулирует доступ к файлу из *внешних* источников или через специальные функции платформы.3 Именно этот компонент является ключевым для понимания проблемы  
  P1.

Эта многоуровневая модель предназначена для гранулярного контроля. Расширенные привилегии по умолчанию отключены, что является основополагающим принципом безопасности, предотвращающим случайное раскрытие данных для внешних систем.3

### **1.2. Привилегия fmxdbc: Технический барьер для ODBC/JDBC**

Для предоставления доступа к данным через ODBC или JDBC используется специальная расширенная привилегия.

* **Определение:** fmxdbc — это системное ключевое слово для расширенной привилегии «Доступ через ODBC/JDBC». Чтобы учетная запись могла успешно подключиться к файлу через драйвер ODBC, ее набор привилегий *обязательно* должен включать эту активированную привилегию.3  
* **Процесс активации:** Включение этой привилегии требует, чтобы администратор, обладающий правом «Управление расширенными привилегиями», вручную открыл файл, зашел в настройки безопасности, отредактировал соответствующий набор привилегий и активировал опцию fmxdbc.3  
* **Анализ:** Проблема клиента заключается в том, что в его файлах отсутствует учетная запись с набором привилегий, где активирована fmxdbc. Задача состоит не просто в том, чтобы «поставить галочку», а в выполнении операции по конфигурированию безопасности, которая требует административных прав высокого уровня для каждого отдельного файла.

### **1.3. Архитектурная первопричина P1: Инкапсуляция на уровне файлов и устаревшие структуры**

Проблема, с которой столкнулся клиент, напрямую вытекает из фундаментальных принципов архитектуры FileMaker.

* **Философия FileMaker:** Каждый файл с расширением .fmp12 представляет собой самодостаточное приложение, инкапсулирующее собственные данные, пользовательский интерфейс, логику (скрипты) и схему безопасности. Политики безопасности не каскадируются автоматически между отдельными файлами.2  
* **Ключевая подсказка — «большое количество файлов»:** Это с высокой вероятностью указывает на устаревшую архитектуру, где до появления FileMaker 7 каждая таблица часто хранилась в отдельном файле. Система клиента, скорее всего, представляет собой коллекцию таких дискретных файлов, а не современное решение с множеством таблиц в одном файле.6  
* **Причинно-следственная связь:**  
  1. **Причина:** Клиент обладает фрагментированным, многофайловым решением FileMaker, что является распространенным архитектурным паттерном для старых систем.  
  2. **Взаимодействие:** Эта архитектура вступает в противоречие с моделью безопасности FileMaker, которая намеренно требует управления безопасностью для внешнего доступа на уровне каждого отдельного файла.  
  3. **Следствие (P1):** В результате возникает проблема, которая масштабируется линейно с количеством файлов. В платформе отсутствует встроенная функция «пакетного включения» ODBC, поскольку это нарушило бы основной принцип инкапсулированной и гранулярной безопасности. Проблема P1 — это результат столкновения архитектурного долга с осознанным решением в области безопасности платформы.

### **1.4. Роль FileMaker Server в хостинге ODBC**

Даже при использовании FileMaker Server, проблема P1 остается актуальной. Для предоставления доступа по ODBC к файлам, размещенным на FileMaker Server, требуется двухэтапный процесс включения:

1. **На уровне сервера:** Коннектор ODBC/JDBC должен быть активирован в консоли администрирования FileMaker Server (Admin Console). Это единый глобальный переключатель для всего сервера.7  
2. **На уровне файла:** Каждый отдельный файл *все равно* должен иметь активированную привилегию fmxdbc для конкретных учетных записей, как было описано выше.7

Даже если клиент использует FileMaker Server и серверный коннектор активен, он все равно столкнется с проблемой P1. Настройка на уровне сервера является необходимым, но не достаточным условием. Это еще раз подтверждает, что узким местом является конфигурация безопасности на уровне каждого отдельного файла.

## **Раздел 2: Путь I \- Прямая автоматизация включения ODBC (скрипты UI)**

### **Цель**

Цель данного раздела — проанализировать осуществимость и риски использования автоматизации пользовательского интерфейса (UI) для прямого решения проблемы P1 путем симуляции ручных шагов, необходимых для включения доступа по ODBC.

### **2.1. Анализ осуществимости: Единственный путь для прямого включения**

Архитектура FileMaker накладывает строгие ограничения на программное изменение настроек безопасности.

* **Ограничение платформы:** Claris не предоставляет нативного шага скрипта или функции API для изменения расширенных привилегий. Это осознанное проектное решение, направленное на предотвращение программной эскалации привилегий и несанкционированного доступа.  
* **«Последний довод»:** Следовательно, единственный возможный способ автоматизировать *точную* задачу включения привилегии fmxdbc для большого количества файлов — это симуляция взаимодействия пользователя с интерфейсом клиента FileMaker Pro.  
* **Инструменты:**  
  * **macOS:** AppleScript является нативным инструментом для этой задачи, способным взаимодействовать с меню, окнами и кнопками.10  
  * **Windows:** AutoHotkey или аналогичные инструменты автоматизации UI являются эквивалентом для данной операционной системы.

### **2.2. План процесса для скрипта автоматизации UI**

Автоматизированный скрипт должен будет выполнять следующую последовательность действий:

1. Получить список всех файлов .fmp12 в указанном каталоге.  
2. Запустить цикл по этому списку.  
3. Для каждого файла в цикле:  
   a. Дать команду FileMaker Pro открыть файл, предоставив учетные данные. Это немедленно создает серьезную проблему: скрипт должен иметь доступ к универсальной учетной записи администратора с паролем, действительной для всех файлов.12

   b. Выполнить навигацию по меню: Файл \-\> Общий доступ \-\> Включить ODBC/JDBC....3

   c. Идентифицировать и нажать на радиокнопку «Вкл.».  
   d. Выбрать файл в списке открытых файлов.  
   e. Выбрать опцию «Указать пользователей по набору привилегий», нажать кнопку «Указать...», отметить необходимые наборы привилегий и нажать «ОК».  
   f. Нажать финальную кнопку «ОК», чтобы закрыть диалоговое окно.  
   g. Закрыть файл.  
4. Повторить для следующего файла в цикле.

### **2.3. Оценка рисков: Хрупкость автоматизации UI**

Подход, основанный на автоматизации UI, сопряжен с высокими рисками, делающими его непригодным для критически важных задач.

  * Точного наименования пунктов меню («Общий доступ», «Включить ODBC/JDBC...»).  
  * Позиции и иерархии элементов UI (например, «флажок 1 строки 3 таблицы 1...» 11).  
  * Системных задержек (появление диалоговых окон, открытие файлов). Часто требуется использование команды delay, но ее значение трудно подобрать, и оно не гарантирует надежности.10  
* **Отсутствие надежной обработки ошибок:** Если диалоговое окно не появится, как ожидалось, или файл не сможет открыться, скрипт, скорее всего, завершится сбоем без возможности чистого восстановления.  
* **Зависимость от версии и ОС:** Обновление FileMaker Pro или базовой операционной системы (macOS/Windows) может легко изменить элементы UI и сломать скрипт.

Этот путь технически возможен, но его рекомендация в качестве основной стратегии для платного клиентского проекта такого масштаба является профессионально безответственной. Это хрупкий обходной путь, а не надежное инженерное решение.

## **Раздел 3: Путь II \- Стратегия нативного экспорта (скрипт Export Records)**

### **Цель**

Цель данного раздела — представить надежную, стабильную и нативную альтернативу, которая полностью обходит проблему ODBC, концентрируясь на истинной цели: извлечении данных.

### **3.1. Глубокий анализ шага скрипта Export Records**

Этот шаг скрипта является основным инструментом FileMaker для экспорта данных и идеально подходит для автоматизации.

* **Основная функциональность:** Это мощный, первоклассный шаг скрипта, разработанный для массового извлечения данных.13  
* **Поддержка автоматизации:** Ключевые опции, такие как С диалогом: Выкл. и возможность указания динамического пути вывода, делают его идеальным для автоматического выполнения без участия пользователя.13  
* **Выполнение на стороне сервера:** Скрипт может быть запущен на FileMaker Server (с помощью шага Perform Script on Server), что значительно быстрее для размещенных файлов, поскольку это позволяет избежать передачи данных по сети на клиентскую машину.15  
* **Ограничения:** Важно учитывать контекст выполнения. Экспорт в формат .fmp12 не поддерживается на сервере, а экспорт в XML не поддерживается в FileMaker Go. Для данного проекта (миграция в PG) эти ограничения не являются критичными.13

### **3.2. Сравнение форматов для импорта в PostgreSQL**

Выбор правильного промежуточного формата данных является ключевым для успешной миграции. Клиентская задача T1 («каковы мои варианты экспорта») решается через анализ следующих опций.

* **CSV (Comma-Separated Values):**  
  * **Преимущества:** Простой, универсально поддерживаемый формат, легко импортируемый в PG с помощью команды COPY.  
  * **Недостатки:** «Глупый» формат, не содержит информации о схеме данных. Могут возникать проблемы с разделителями внутри данных, кодировкой символов и переносами строк. Стандартный экспорт из FileMaker оборачивает каждое поле в кавычки, что безопасно, но может потребовать дополнительной обработки.16  
* **XML (Extensible Markup Language):**  
  * **Преимущества:** Самоописываемый формат (схема может быть включена). FileMaker может экспортировать в двух грамматиках (FMPDSORESULT и FMPXMLRESULT) и применять XSLT-преобразование «на лету», что обеспечивает огромную гибкость.13 Это можно использовать для прямой генерации  
    INSERT-запросов или более структурированного формата.  
  * **Недостатки:** Более многословный, требует более сложного парсинга на стороне PG по сравнению с COPY из CSV.  
* **XLSX (Excel):**  
  * **Преимущества:** Может сохранять некоторое форматирование данных.  
  * **Недостатки:** Бинарный формат, который добавляет ненужную промежуточную зависимость (требует библиотеку для парсинга XLSX на стороне PG). Не рекомендуется для миграции между серверами.13

Для обеспечения ясной основы для принятия решений, приведена следующая сравнительная таблица.

**Таблица 3.2.1: Сравнение форматов экспорта для целостности данных и сложности импорта в PostgreSQL**

| Формат | Типизация данных | Информация о схеме | Обработка спецсимволов | Метод импорта в PG | Сложность | Рекомендация |
| :---- | :---- | :---- | :---- | :---- | :---- | :---- |
| **CSV** | Слабая (все как текст) | Отсутствует | Надежно (с кавычками) | COPY FROM | Низкая | **Рекомендуется** |
| **XML** | Строгая (через схему) | Встроенная | Надежно | xmlparse, кастомный парсер | Высокая | Для сложных преобразований |

Эта таблица предоставляет четкую структуру для выбора оптимального промежуточного формата, напрямую связывая возможности FileMaker с требованиями целевой системы, PostgreSQL.

### **3.3. Архитектура автоматизированного решения для экспорта: Подход с «мастер-скриптом»**

Для управления процессом экспорта из множества файлов наиболее надежным является модульный подход.

* **Концепция:** Создание одного «управляющего» файла FileMaker или внешнего скрипта (например, Python, Shell), который организует весь процесс.  
* **План процесса:**  
  1. Управляющий скрипт получает список всех целевых файлов .fmp12.  
  2. Он запускает цикл по этому списку. Для каждого файла:  
  3. Он открывает файл. Это требует наличия универсальных учетных данных администратора — то же критическое требование, что и в Пути I.  
  4. Он вызывает стандартизированный скрипт внутри целевого файла (например, \_utility\_exportAllTables) с помощью шага Perform Script.18  
  5. Скрипт \_utility\_exportAllTables содержит логику Export Records для каждой таблицы в этом файле, экспортируя данные в динамически названный CSV или XML файл (например, \[ИмяФайла\]\_\[ИмяТаблицы\].csv).  
  6. Управляющий скрипт закрывает файл и переходит к следующему.

Эта архитектура является модульной, поддерживаемой и высоконадежной. Она использует предназначенные для автоматизации функции платформы. Основные усилия по разработке заключаются в создании стандартизированного внутреннего скрипта и внешнего управляющего скрипта.

### **3.4. Обработка сложных данных (решение задачи T2)**

Простая миграция данных редко бывает достаточной. Необходимо учесть специфические для FileMaker типы данных, что напрямую решает задачу клиента T2 («как выбрать информацию для экспорта таблиц»).

* **Поля-контейнеры:** Эти поля хранят файлы (изображения, PDF и т.д.). Шаг Export Records не экспортирует бинарные данные. Правильный метод — получить путь к внешне хранящемуся файлу контейнера с помощью функции GetContainerAttribute(поле; "externalFiles"), а затем скопировать сам файл. CSV/XML будет содержать относительный путь к файлу.21 Это критический нюанс для полной миграции.  
* **Повторяющиеся поля:** Устаревшая конструкция FileMaker. Они должны быть нормализованы в отдельные записи в PG. Экспорт в XML с XSLT-преобразованием — самый элегантный способ сделать это «на лету».22 Альтернативно, скрипт может циклически проходить по повторениям и создавать новую строку во временной экспортной таблице перед экспортом.  
* **Кодировка символов:** Диалоговое окно Export Records позволяет указать кодировку вывода (например, UTF-8, UTF-16). Для PG стандартом является UTF-8, и ее следует явно установить для обеспечения совместимости.17

## **Раздел 4: Путь III \- Стратегия «API-First» (FileMaker Data API)**

### **Цель**

Цель данного раздела — оценить самый современный, но и самый требовательный метод извлечения данных.

### **4.1. Технический обзор: REST, JSON и Data API**

FileMaker Data API представляет собой современный интерфейс для взаимодействия с данными.

* **Механизм:** Data API — это RESTful-интерфейс, который позволяет внешним приложениям выполнять операции CRUD (создание, чтение, обновление, удаление) и запускать скрипты через стандартные HTTPS-запросы. Данные обмениваются в формате JSON.24  
* **Преимущества:** Не требует драйверов (не нужно устанавливать/настраивать драйверы ODBC/JDBC), не зависит от платформы и использует веб\-стандарты.27

### **4.2. Предварительные требования и архитектурное влияние**

Использование Data API накладывает строгие требования на среду хостинга и конфигурацию безопасности.

* **FileMaker Server обязателен:** Data API является функцией FileMaker Server или FileMaker Cloud. Его нельзя использовать с локальными файлами, открытыми в FileMaker Pro. Это серьезное архитектурное ограничение. Описание клиента не уточняет его среду хостинга, и это первый вопрос, который необходимо задать.25  
* **Та же проблема, что и P1:** Так же, как и ODBC, Data API требует включения расширенной привилегии на уровне каждого файла и каждого набора привилегий: fmrest.25

На первый взгляд, этот путь имеет тот же фатальный недостаток, что и ODBC. Однако это приводит к важному стратегическому выводу. Если предположить, что для получения любого внешнего доступа необходимо однократно применить хрупкий UI-скрипт (Путь I), то возникает вопрос: какую привилегию следует включить этим скриптом? fmxdbc (для ODBC) или fmrest (для Data API)? Data API является технологически превосходящим выбором для фактического извлечения данных: он быстрее, гибче (JSON богаче, чем CSV) и не требует драйверов.

Следовательно, если файлы находятся на сервере, оптимальный (хотя и сложный) путь таков: **использовать временный UI-скрипт для однократного включения fmrest во всех файлах, а затем использовать надежный клиент (например, на Python или с помощью cURL) для извлечения данных через Data API.** Этот гибридный подход использует «хакерское» решение для наименьшей возможной части проблемы и «инженерное» решение для основной передачи данных.

### **4.3. План процесса: Создание клиента для миграции**

Клиент для миграции будет представлять собой скрипт (например, на Python, Node.js), который:

1. Аутентифицируется на FileMaker Server для получения токена сессии.  
2. Итерирует по списку имен баз данных.  
3. Для каждой базы данных запрашивает макеты (которые соответствуют таблицам), чтобы получить записи в формате JSON.  
4. Обрабатывает пагинацию для больших наборов записей.  
5. Парсит JSON и загружает его в PostgreSQL.

### **4.4. OData как альтернативный API**

FileMaker Server также поддерживает OData, еще один стандарт RESTful API.27 Ключевое преимущество OData заключается в том, что это межплатформенный стандарт, который даже может использоваться для изменения схемы (что нерелевантно для данной миграции).27 Для задачи чистого извлечения данных нативный FileMaker Data API более тесно интегрирован и, как правило, достаточен.27

## **Раздел 5: Путь IV \- Использование сторонних инструментов для миграции**

### **Цель**

Цель данного раздела — оценить, может ли готовое («buy») решение решить проблему быстрее, чем разработанное на заказ («build»).

### **5.1. Обзор рынка: Инструменты как ускорители**

На рынке существуют специализированные инструменты, предназначенные для упрощения миграции между различными СУБД.

* **Ключевые игроки:** В исследованиях упоминаются такие инструменты, как FmPro Migrator 22 и Full Convert.33 Компания CData также предоставляет драйверы, облегчающие подключение.34  
* **Основное ценностное предложение:** Их главная функция — автоматизация утомительного процесса трансляции схемы (сопоставление типов полей FileMaker с типами PostgreSQL) и передачи данных.23

### **5.2. Критическая зависимость: Предварительное требование ODBC**

Эти инструменты не являются волшебной палочкой и имеют те же ограничения, что и другие методы внешнего доступа.

* **«Уловка-22»:** Документация показывает, что для подключения к исходной базе данных эти инструменты обычно полагаются на предварительно настроенное ODBC-соединение.23  
* **Анализ:** Это означает, что для данного клиента эти инструменты не могут быть использованы «из коробки». Клиент должен *сначала* решить проблему P1, используя один из других путей (например, UI-скриптинг для включения ODBC), прежде чем инструмент вроде FmPro Migrator сможет даже подключиться к файлам.

### **5.3. Автоматическая трансляция схемы и типов данных**

Несмотря на зависимость от ODBC, эти инструменты предлагают полезные функции автоматизации.

* **Функциональность:** Они интеллектуально решают такие проблемы, как:  
  * Преобразование общего типа Number из FileMaker в INTEGER или FLOAT в PG.6  
  * Сопоставление больших текстовых полей с типом TEXT.23  
  * Обработка недопустимых символов в именах полей.23  
  * Попытки преобразования полей-контейнеров и повторяющихся полей.23  
* **Ограничения:** Преобразование не является идеальным. Специфичные для FileMaker конструкции, такие как итоговые поля, вычисляемые поля и скрипты, не могут быть напрямую перенесены и должны быть реализованы заново в экосистеме PG (например, в виде представлений или логики приложения).6

### **5.4. Анализ затрат и выгод: «Build vs. Buy»**

Выбор между разработкой собственного решения и покупкой готового инструмента требует взвешенного подхода.

* **Build (Путь II/III):** Более высокие начальные затраты на разработку. Полный контроль над процессом. Отсутствие лицензионных отчислений.  
* **Buy (Путь IV):** Лицензионные затраты. Потенциально быстрее на этапе сопоставления схемы/данных. Меньше контроля над процессом. **Критически важно, что это не устраняет необходимости сначала решить P1.**  
* **Вердикт:** Для данного проекта опция «buy» не является полным решением. Это опциональное дополнение *после* того, как доступ будет установлен. Учитывая необходимость в любом случае писать кастомный скрипт для решения P1, расширение этого скрипта для обработки экспорта (Путь II), скорее всего, будет более эффективным, чем гибридный подход.

## **Раздел 7: Дополнительные проблемы и продвинутые соображения**

### **Цель**

Цель данного раздела — рассмотреть вторичные проблемы и нюансы, упомянутые клиентом и выявленные в ходе исследования.

### **7.1. Проблема «повторного входа»: Критичность единой аутентификации**

Замечание клиента о том, что «может потребоваться повторный вход в систему с именем пользователя и паролем», является серьезным сигналом о риске. В документации рекомендуется использовать идентичные учетные записи, чтобы избежать диалоговых окон аутентификации при доступе к связанным файлам.2 Если каждый из «большого количества файлов» имеет уникальный пароль, любой автоматизированный циклический скрипт потерпит неудачу на первом же шаге.

Это самый большой риск для осуществимости проекта. Первоначальная консультация *должна* быть сосредоточена на этом вопросе. Если учетные данные не унифицированы, может потребоваться предварительный (и, возможно, ручной) проект по стандартизации административной учетной записи во всех файлах, прежде чем можно будет начать любую техническую работу по миграции.

### **7.2. Инструмент миграции данных FileMaker (DMT): Вводящее в заблуждение название**

Важно не путать общую задачу миграции с конкретным инструментом от Claris.

* **Функция:** DMT — это инструмент командной строки, предназначенный для миграции *данных* из старой версии файла FileMaker в новую, структурно отличающуюся версию («клон») *того же* решения.35  
* **Сценарий использования:** Он предназначен для разработчиков, развертывающих новые версии своих приложений, а не для миграции на другую платформу баз данных, такую как PostgreSQL.36  
* **Ограничения:** Он не переносит внешне хранимые данные контейнеров и имеет другие специфические требования.37  
* **Вывод:** DMT абсолютно нерелевантен для проблемы данного клиента. Его название может сбить с толку, но это инструмент для обновлений FileMaker-to-FileMaker, а не для миграции FileMaker-to-SQL. Этот раздел явно проясняет это, чтобы предотвратить движение по ложному пути.

### **7.3. Обеспечение целостности данных: Больше, чем просто экспорт**

Миграция — это не только перенос строк и столбцов.

* **Вычисления и итоги:** Вычисляемые и итоговые поля в FileMaker — это не реальные данные; они вычисляются «на лету». Их нельзя «экспортировать». Они должны быть воссозданы в PostgreSQL в виде VIEW или как логика в прикладном уровне, который будет использовать новую базу данных.6  
* **Сопоставление данных:** План миграции должен включать подробный словарь данных, сопоставляющий каждое поле FileMaker с соответствующим столбцом PostgreSQL с правильным типом данных (TEXT, VARCHAR, INTEGER, NUMERIC, TIMESTAMP, BYTEA для контейнеров).6

Помни: я пишу клиенту или потенциальному клиенту.
Делать в любом случае буду я, а не клиент.
Вместо «you need» используй 2 альтернативы:
#### 30.8.1.
Нейтральные фразы типа «it is necessary».
#### 30.8.2.
Глаголы в неопределённой форме.
Например, во фрагменте ниже использованы подобные глаголы «set up», «create»:
```
1.2) Set up the transfer of login events from WordPress to Power BI using Fabric / OneLake.
1.2.1) Set up a «Data Pipeline» from the WordPress database table that stores login events (see point 1.1) to Fabric / OneLake.
1.2.2) Set up a connection from Power BI to Fabric / OneLake to pass login events.
1.3) Create the data model in Power BI.
```
Обрати внимание, в этом фрагменте не говорится, кто именно будет выполнять описанные действия: ответственность не перекладывается на клиента, в отличие от «you need».

### 30.9.
Каждый шаг `A1` нумеруй.
Например: `1)`.

### 30.10.
Для вложенных шагов используй вложенную нумерацию. 
Например: `1.1)`.

### 30.11.
Для каждого шага `A1` указывай конкретное обоснование из всего задокументированного в моём запросе анализа проекта клиента.

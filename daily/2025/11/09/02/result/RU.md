Your misconceptions:
1) Elastic Query (далее — `EQ`) does not support Azure Managed Identity (далее — `AMI`). 
Это наиболее фатальный технический недостаток, который делает предложенную вами архитектуру dead on arrival.
Официальная документация Microsoft по `EQ` содержит недвусмысленное и фатальное ограничение: «Elastic queries only supported when connecting with SQL Server Authentication».
Это ограничение напрямую противоречит всем современным лучшим практикам безопасности в Azure, которые настоятельно предписывают использовать аутентификацию Microsoft Entra ID (далее — `EID`) и `AMI` именно для того, чтобы избежать использования и хранения паролей.
2) Вы просите у Eccovia (далее — `E`) (стороннего SaaS-вендора) прямой SQL-доступ.
Из-за пункта 1, вы вынуждены запрашивать у `E` не `AMI`, а SQL-логин и пароль.
Ни один компетентный SaaS-вендор, особенно в сфере здравоохранения, соответствующей HIPAA, в 2025 году никогда не предоставит внешнему субъекту SQL-логин и пароль к своей производственной многопользовательской базе данных. 
Это было бы равносильно передаче ключей от всего хранилища данных.
3) Ваша архитектура — это интеграция point-to-point: (Собственная Azure SQL DB) ⟷ `E`. 
Предписанная архитектура EQ — это one-to-many внутри собственной среды клиента: (Главная DB) ⟷ (Шард 1, Шард 2,..., Шард N).
Это фундаментальное несоответствие. 
`EQ` не является универсальной заменой связанных серверов и не предназначен для подключения к произвольным внешним базам данных SQL, особенно к изолированной, защищенной PaaS-базе данных, управляемой сторонним SaaS-поставщиком.
4) Microsoft по-прежнему помечает `EQ` как «in preview».
`EQ` «находится в статусе preview уже несколько лет».
Статус «Preview» имеет конкретные и серьезные последствия: отсутствие Соглашения об уровне обслуживания (SLA) и отсутствие полной производственной поддержки. 
Вы работаете в секторе здравоохранения, обрабатывая Protected Health Information (PHI). 
Это среда с максимально высокими требованиями к стабильности, безопасности и поддержке. 
Предложение построить основную производственную аналитическую платформу на технологии, не имеющей SLA  и находящейся в состоянии бессрочной предварительной версии, является грубым нарушением базовых принципов управления техническими рисками.
5) Заблуждение о «hesitance» `E`
`E`, как поставщик EHR-системы, обрабатывающей данные о психическом здоровье, юридически классифицируется как «Business Associate» в соответствии HIPAA. 
В этом качестве `E` несет прямую юридическую ответственность за обеспечение конфиденциальности, целостности и доступности Protected Health Information (`PHI`).
Даже read-only доступ может раскрыть конфиденциальные данные, если учетные данные будут использованы не по назначению.
В секторе здравоохранения или при работе с любыми конфиденциальными данными, несанкционированное чтение (раскрытие) часто является более серьезным нарушением, чем несанкционированная запись (изменение).
6) Заблуждение о «viewer-level access» (далее — `VLA`)
6.1) В контексте SQL Server, `VLA` соответствует встроенной роли базы данных `db_datareader`.
 Эта роль предоставляет только разрешение `SELECT` на все пользовательские таблицы. 
Она не предоставляет никаких разрешений Data Definition Language (`DDL`).
6.2) Для реализации `EQ` вам нужно выполнить T-SQL команду `CREATE EXTERNAL DATA SOURCE` (далее — `CEDS`).
Для выполнения `CEDS` требуется разрешение `ALTER ANY EXTERNAL DATA SOURCE` (далее — `AAEDS`).
6.3)
`AAEDS` — это высокопривилегированное разрешение DDL. 
Попытка выполнить `CEDS` с правами `db_datareader` завершится ошибкой «PERMISSION DENIED».
6.4) `AAEDS` предоставляет возможность доступа ко всем database scoped credentials.
Официальная документация Microsoft пишет: это разрешение должно рассматриваться как высокопривилегированное (highly privileged) и должно предоставляться только доверенным принципалам в системе». 
6.5) По сути, `E` вам вежливо отказали, предложив вам только `VLA`, а вы приняли этот отказ за работающий компромисс.

1) Надо убедиться что ни одно из ваших приложений не зависит от механизма 8.3 (коротких) имён. 
2) Надо создать резервную копию критически важной информации.   
3) Надо выполнить команду:
```
fsutil 8dot3name query
```
https://learn.microsoft.com/en-us/windows-server/administration/windows-commands/fsutil-8dot3name
3.1) Если программа `fsutil` ответит, что `NtfsDisable8dot3NameCreation` равно `1` (или `3`), значит короткие имена уже отключены (полностью или частично). 
3.2) Если программа `fsutil` ответит `0` — значит, короткие имена включены. 
3.3) Если программа `fsutil` ответит `2` — значит, использется настройка на уровне томов.
4) В случае 3.2 8.3-имена надо отключить.
4.1) Проще всего это сделать той же программой `fsutil`:
```
fsutil 8dot3name set 1
```
4.2) Другим способом это можно сделать через «Registry Editor»:
4.2.1) Открыть «Registry Editor» (через «Start» → «Run» → введите `regedit`).   
4.2.2) Перейти по пути: «HKEY_LOCAL_MACHINE» → «SYSTEM» → «CurrentControlSet» → «Control» → «FileSystem».   
4.2.3) Найти параметр `NtfsDisable8dot3NameCreation`.   
4.2.4) Изменить «Value data» на 1, чтобы отключить механизм 8.3 для всех томов.   
4.2.5) Закрыть «Registry Editor».
5) Рекомендую перезагрузить сервер, чтобы изменения вступили в силу для всех новых операций с файлами.   
После перезагрузки новые файлы и каталоги не будут получать короткие имена. 
Однако для уже существующих файлов короткие имена по-прежнему могут сохраняться.
6) Удаление уже существующих 8.3-имен.
Это особенно важно для каталога с веб-контентом IIS.
6.1) Надо выполнить команду:
```
fsutil 8dot3name strip
```
6.1.1) Пример (с включением обработки подкаталогов и записи лога в `strip-log.txt`):
```
fsutil 8dot3name strip /s /l strip-log.txt D:\inetpub\wwwroot
```
В этой команде:
- `/s` указывает применение ко всем подкаталогам.  
- `/l` позволяет указать файл лога; в данном случае `strip-log.txt`.   
- `D:\inetpub\wwwroot` — это пример пути к каталогу, где находится веб-контент. 
6.2) Надо проанализировать полученный лог (`strip-log.txt` из примера 6.1.1). 
Если какие-либо приложения всё же ссылаются на короткие имена, удаление 8.3-имен может привести к ошибкам в этих приложениях. 
Поэтому перед окончательным удалением можно запустить команду с параметром `/t` (тестовый режим), чтобы определить потенциально проблемные файлы.
7) После внесённых изменений и (при необходимости) удаления старых коротких имён повторно запустите то средство сканирования, которое зафиксировало уязвимость IIS Short Name Enumeration. 
Обычно это происходит через внешний PCI DSS Approved Scanning Vendor (ASV) или с помощью программного обеспечения (например, Tenable Nessus, Qualys или др.).  
Убедитесь, что проблема более не отображается в отчёте сканирования.
8) Дополнительные меры (по необходимости)
8.1. Если у вас есть старые приложения, которым требуются короткие имена, можно вместо глобального отключения 8.3 (значение `1`) установить `NtfsDisable8dot3NameCreation=2` или `3`. 
После этого вы сможете вручную включать или отключать 8.3-имена для отдельных томов.   
8.2. Можно настроить фильтрацию символа `~` в Microsoft Internet Information Services (IIS Manager) → «Request Filtering» или через «URL Rewrite» для блокировки специфических запросов, но это решение будет только скрывать уязвимость на уровне IIS. 
Если цель — именно выполнить требования PCI DSS, чаще всего рекомендуется полноценное отключение 8.3 (с последующим удалением существующих коротких имён).
9) Обычно надо подготовить отчёт для аудиторов PCI DSS с указанием, какие конкретные шаги были предприняты.

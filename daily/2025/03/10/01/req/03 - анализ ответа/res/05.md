## 1. Несоответствия в IP-адресации (Static Route vs. IP Monitoring)

### 1.1 Статический маршрут
Вы указали:
```
set routing-options static route 0.0.0.0/0 qualified-next-hop 10.0.0.253 preference 10
set routing-options static route 0.0.0.0/0 next-hop pp0.0 preference 20
```
То есть основным шлюзом (preference 10) является `10.0.0.253`, а резервным (preference 20) — интерфейс `pp0.0`.

### 1.2 IP Monitoring
Далее в `ip-monitoring` вы ссылаетесь на **другой** next-hop:
```
set services ip-monitoring policy MAIN-LINK then preferred-route route 0.0.0.0/0 next-hop 10.0.0.254
set services ip-monitoring policy MAIN-LINK then preferred-route route 0.0.0.0/0 qualified-next-hop pp0.0
set services ip-monitoring policy MAIN-LINK then failover-route route 0.0.0.0/0 next-hop pp0.0
```
Здесь «предпочитаемый» шлюз указан как `10.0.0.254`, хотя выше по статике вы пишете `10.0.0.253`. Если IP `10.0.0.254/30` — **ваш** адрес на reth1, то роутер провайдера обычно `10.0.0.253`. Соответственно, **либо** в статике надо использовать `.254`, **либо** в `ip-monitoring` ссылаться на `.253`. Важно, чтобы основной next-hop был один и тот же (то есть чтобы совпадал “живой” шлюз и то, что проверяется/меняется в route при failover).

Кроме того, если вы _одновременно_ используете и статический маршрут с разными preference, _и_ ещё `ip-monitoring`, который «вживую» правит маршрут 0.0.0.0/0, могут возникать конфликты (дублирование «кто из них правит»). Обычно делают один из двух вариантов:

1. **Простой вариант**: статический маршрут с разными `preference` (10 на основной, 20 на резервный) — без ip-monitoring. При пропадании линка интерфейс в состоянии `down`, и JunOS автоматически переключится на резерв. Минус: если интерфейс физически «up», но трафик не ходит (проблемы у провайдера), автоматического переключения может не быть.

2. **«Умный» вариант с ip-monitoring**: прописать изначально только _один_ маршрут для основного next-hop (например, preference 10) и давать ip-monitoring «убирать» или «включать» альтернативную 0.0.0.0/0 через 4G. Тогда при отсутствии ответа от probe JunOS удаляет основной маршрут или добавляет резервный.

Главное — определиться, какой сценарий вам нужен, и чтобы IP-адреса next-hop совпадали.

## 2. Автоматическое переключение DNS (event-options)

Само по себе решение через `event-options` с «delete system services dns forwarding server …» и «set system services dns forwarding server …» — рабочее, но довольно «экзотическое»:
1. В реальной жизни часто используют _один_ набор публичных DNS (например, 8.8.8.8/8.8.4.4), которые доступны и через основной канал, и через 4G.  
2. Если 4G-провайдер действительно требует **только** свои DNS, тогда ваш подход оправдан. Но стоит понимать, что скриптовое изменение конфигурации по событию `chassiscluster_status_change` требует аккуратного теста.  

Обратите внимание, что в `match "Redundancy group 1: node0: Lost primary status"` может потребоваться точная строка из `show log messages` — иногда формулировка события может отличаться в зависимости от версии JunOS. Точно так же с `Gained primary status`.

## 3. Разница между reth-интерфейсами и ae-интерфейсами

Вы в некоторых местах (п.18) пишете:
```
show interfaces ae0 extensive
```
Но при этом у вас выше используется:
```
set interfaces reth0 ...
set interfaces reth1 ...
```
На SRX в кластере действительно работают **rethX** (redundant ethernet interfaces).  
- **aeX** обычно встречается на EX/MX/в standalone-режиме или когда SRX используют обычные LACP-«транки» без HA.  

Если у вас включён _именно_ «MC-LAG» на SRX-кластере (multi-chassis LAG), то “reth” и “lacp active” могут сочетаться (это называется «redundant LAG»), однако **show interfaces ae0 extensive** при этом не покажет нужный интерфейс, поскольку логически он называется `reth0`/`reth1`. Для просмотра LAG-статуса на SRX-кластере обычно используют:
```
show interfaces reth0
show lacp interfaces reth0
```
или  
```
show chassis cluster interfaces
```
и т. д.

## 4. Порядок настройки Chassis Cluster

Общий алгоритм настройки кластера SRX (особенно если устройства «с нуля») может потребовать чуть более строгой «последовательности»:

1. На **node0** (зайдя в CLI):
   ```bash
   set chassis cluster cluster-id 1 node 0
   set chassis cluster control-link ge-0/0/x
   set chassis cluster fabric-link ge-0/0/y
   commit check
   commit
   request system reboot
   ```
2. После перезагрузки **node0**, на **node1**:
   ```bash
   set chassis cluster cluster-id 1 node 1
   set chassis cluster control-link ge-1/0/x
   set chassis cluster fabric-link ge-1/0/y
   commit check
   commit
   request system reboot
   ```
3. После этого они сами сформируют кластер.  

У вас всё в целом описано, просто имейте в виду, что «одним списком» такие команды обычно не применяются — нужно перезагрузить сначала node0, потом node1.

## 5. Пара мелочей по PPP (4G) на SRX300

- Для **внешнего** 4G-модема через USB на младших SRX обычно действительно поднимается интерфейс `pp0` (либо `pp0.0`), и нужна `access profile`, CHAP/PAP, и т. д.  
- У некоторых моделей SRX300 Series есть встроенный LTE-модуль, тогда интерфейс может быть `ge-0/0/x` (с `wireless` опциями) или `lt-0/0/0` — но это уже зависит от конкретного SKU.  

Проверьте, что в вашем случае устройство действительно видит PPP-интерфейс и вы правильно указали методы аутентификации.

## 6. NAT и Security Policies

В конце вы добавляете «универсальную» политику:
```
set security policies from-zone TRUST to-zone UNTRUST policy ALLOW-ALL ...
```
и исходящий NAT:
```
set security nat source rule-set RS1 from zone TRUST to zone UNTRUST
set security nat source rule-set RS1 rule R1 match source-address 192.168.1.0/24
set security nat source rule-set RS1 rule R1 then source-nat interface
```
Если у вас **оба** интерфейса (reth1.0 и pp0.0) находятся в одной и той же «UNTRUST»-зоне, то этого будет достаточно и для основного WAN, и для 4G (при фейловере). Всё выглядит логично. Главное — не забыть добавить `pp0.0` в зону UNTRUST:
```
set security zones security-zone UNTRUST interfaces pp0.0 host-inbound-traffic system-services ping
...
```
Что вы, судя по всему, уже сделали.

## 7. Нужно ли что-то ещё?

### 7.1 Мониторинг RG0 vs RG1
- Вы мониторите доступность основного WAN через `ip-monitoring` в RG1 (похоже, именно RG1 «возит» ваш reth1).  
- Для полного отказоустойчивого сценария бывает, что ещё включают `RG0` (Routing Engine redundancy) или мониторят интерфейс RG0, чтобы при пропадании control/fabric не было «split-brain». Но это уже деталь, зависящая от того, **какой** уровень отказоустойчивости требуется.

### 7.2 Preempt (автоматический возврат)
- Если вы хотите, чтобы при восстановлении **node0** автоматом снова становился «primary» по RG1, обычно включают `preempt`. В JunOS для RG > 0 по умолчанию preempt включён, но иногда админы отключают. Убедитесь, что этот момент вас устраивает.

### 7.3 Синтаксис event-options
- Убедитесь, что у вас всё верно с контекстом (особенно команды вида `delete system services dns forwarding server x.x.x.x`). Иногда нужно перед ними «rollback 0», либо нужно заходить в иерархию `edit system services dns forwarding`; а также нужно указывать `commit confirmed 1` или что-то подобное.  
- Часто вместо `event-options policy … then execute-commands` используют event-script на SLAX/PyEZ — более гибко, но чуть сложнее.  

Если коротко: **идея верная**, но в реальности без тестирования такие «динамические команды» порой не сходу проходят. Так что обязательно проверяйте через логи, действительно ли команда успешно применяется.

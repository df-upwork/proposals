## **2. Архитектурный Ландшафт Resware**

### **2.1. Историческое Наследие и Эволюция WCF к REST**

Для глубокого понимания текущего состояния Resware API необходимо рассмотреть его технологический генезис. Resware, будучи зрелым продуктом на базе технологий Microsoft (.NET), исторически опирался на Windows Communication Foundation (WCF) для организации межсервисного взаимодействия. WCF — это мощный, но тяжеловесный фреймворк, ориентированный преимущественно на SOAP (Simple Object Access Protocol) и бинарные протоколы обмена данными.6

SOAP, как протокол, обладает строгой типизацией и встроенными механизмами безопасности (WS-Security), что делало его стандартом де-факто для энтерпрайз-систем начала 2000-х годов.5 Однако, жесткость контрактов XML и сложность парсинга сообщений создавали высокий порог входа для интеграции с веб-сервисами и мобильными приложениями. В ответ на требования рынка, Adeptive Software начала внедрять RESTful-интерфейсы, которые работают поверх существующего стека, предоставляя более легковесный доступ к функциональности системы.7

Текущая архитектура Resware API может быть охарактеризована как **гибридная**. Она не является "чистым" REST API в академическом понимании (как, например, GitHub API), а представляет собой прагматичную надстройку над внутренней бизнес-логикой. Это подтверждается тем фактом, что для активации определенных возможностей REST API (например, аутентификации) часто требуется включение "Basic Authentication for WCF" на уровне конфигурации сервера.6 Это свидетельствует о том, что под капотом REST-эндпоинты транслируются в вызовы внутренних сервисов WCF.

### **2.2. Дуализм Протоколов: Сосуществование XMLORDER и REST**

Уникальной чертой интеграционной среды Resware является параллельное использование двух различных парадигм обмена данными:

1. RPC-style XML (eService/XMLORDER):  
   Многие критические функции, такие как создание новых заказов (Order Placement), исторически реализуются через отправку XML-пакетов на единый эндпоинт (часто вида eService/?mode=XMLORDER... или аналогичный). Этот подход напоминает удаленный вызов процедур (RPC), где действие определяется не HTTP-методом, а содержимым или параметрами URL.4 Структура XML здесь чрезвычайно детализирована и строго регламентирована схемами Resware.  
2. Ресурсо-ориентированный REST (Api/files/...):  
   Более современные функции, особенно связанные с документооборотом и взаимодействием с внешними платформами (например, RON), реализуются через интуитивно понятные REST-маршруты. Здесь используются стандартные HTTP-глаголы (POST, GET) для манипуляции ресурсами: файлами, документами, заметками.3

Эта двойственность требует от разработчиков интеграций гибкости: для инициации процесса может потребоваться формирование сложного XML-документа, а для последующего отслеживания статуса и загрузки результатов — использование JSON-запросов к REST API.

### **2.3. Влияние Экосистемы Qualia и Resware 10**

Приобретение Adeptive Software компанией Qualia внесло существенные коррективы в дорожную карту развития API. Qualia активно продвигает унификацию интерфейсов, стремясь к созданию единого стандарта взаимодействия для всех своих продуктов (Core, Atlas, Connect, Resware).

**Resware 10** стал вехой в этом процессе. Хотя фундаментальная логика API сохранилась для обеспечения обратной совместимости, инфраструктурный слой претерпел изменения.9 Внедрение новых инженерных практик и CI/CD пайплайнов в разработку Resware привело к повышению стабильности и производительности API.10 Заявленное ускорение рабочих процессов на 21% 10 напрямую влияет на время отклика синхронных API-запросов, что критично для высоконагруженных интеграций.

Кроме того, Qualia вводит концепцию **Qualia API** на базе GraphQL.11 Это представляет собой следующий эволюционный шаг — декларативный API, позволяющий клиентам запрашивать точно определенные наборы данных. Однако, важно различать "нативный" Resware REST API, который используется для низкоуровневых интеграций on-premise инсталляций, и облачный Qualia API, который, вероятно, служит фасадом для облачных компонентов экосистемы.

## **3. Модель Безопасности и Аутентификации**

В индустрии, регулируемой строгими стандартами защиты данных (GLBA, ALTA Best Practices), безопасность API не может быть второстепенной задачей. Resware реализует комплексную модель защиты, требующую правильной конфигурации как на сетевом, так и на прикладном уровне.

### **3.1. Механизмы Аутентификации**

Анализ документации выявляет несколько методов аутентификации, применяемых в зависимости от типа интеграции и конфигурации партнера.

#### **3.1.1. Basic Authentication (HTTP Basic)**

Это наиболее распространенный метод для REST-взаимодействия. Клиентское приложение передает учетные данные (имя пользователя и пароль) в заголовке Authorization, закодированном в Base64 (Basic <base64_credentials>).

* **Контекст использования:** Широко применяется в интеграциях типа "сервер-сервер", например, с платформами Notarize/Proof.12  
* **Требования к конфигурации:** Критически важным нюансом является то, что поддержка Basic Auth для сервисов WCF часто отключена по умолчанию в on-premise инсталляциях Resware. Администраторы должны явно запросить у поддержки Adeptive включение опции "Basic Authentication for WCF".6 Без этого шага API будет отклонять запросы, ожидая аутентификацию по проприетарному протоколу Microsoft NTLM или Kerberos, что затрудняет интеграцию с не-Windows системами.

#### **3.1.2. Subscription Keys (API Keys)**

В некоторых сценариях, особенно при работе через партнерские шлюзы или специфические сервисы (например, Notaroo), используется механизм API-ключей.

* **Механизм:** Ключ передается как параметр строки запроса (Query String Parameter), например: subscription-key={YourSubscriptionKey}.6  
* **Преимущества:** Этот метод позволяет отделить аутентификацию приложения от аутентификации пользователя, упрощая ротацию ключей и отзыв доступа без смены паролей пользователей.

#### **3.1.3. Контекстные Идентификаторы (Tenant Isolation)**

Помимо подтверждения личности вызывающей стороны, API требует идентификации контекста выполнения. Поскольку Resware поддерживает мульти-офисные конфигурации, запросы часто должны содержать:

* Office ID: Идентификатор конкретного офиса или филиала титульной компании.  
* Client ID: Идентификатор клиента внутри системы вендора.  
  Эти параметры могут передаваться как часть URL, в теле запроса или через кастомные заголовки, в зависимости от конкретного эндпоинта.4

### **3.2. Ролевая Модель Доступа (RBAC)**

Успешная аутентификация — это лишь первый шаг. Система авторизации Resware базируется на детальной матрице ролей. Для API-пользователя (обычно это служебная учетная запись "Partner Employee") должны быть явно активированы специфические права в разделе "Website Roles" административной панели.4

**Таблица 1. Критические роли для работы с API**

| Название Роли (Website Role) | Функциональное Назначение |
| :---- | :---- |
| **Web Services: Access All Files** | Предоставляет глобальный доступ к файлам для системных интеграций (Resware-to-Resware). Критично для сервисных аккаунтов. |
| **Web Services: Search Files** | Позволяет выполнять поиск файлов по номеру или другим критериям. Необходим для первого шага большинства интеграций. |
| **Web Services: Add Documents** | Разрешает загрузку файлов (PDF, TIFF и др.) в систему. |
| **Web Services: Get Documents** | Разрешает скачивание документов для внешней обработки (например, отправки на подпись). |
| **Web Services: Add Notes** | Позволяет создавать заметки, которые часто служат триггерами для внутренних процессов. |
| **Web Services: Get Custom Fields** | Доступ к чтению пользовательских полей, необходимых для специфических данных сделки.13 |
| **Web Services: Order Placement** | Право на создание новых заказов в системе. |

Отсутствие любой из этих ролей приведет к ошибкам доступа (HTTP 403 Forbidden), даже если логин и пароль верны. Это частая причина проблем при первоначальной настройке интеграции.

### **3.3. Сетевая Безопасность и Шифрование**

Учитывая передачу чувствительных данных (Social Security Numbers, финансовые детали), все взаимодействие с API должно происходить исключительно по протоколу **HTTPS** с использованием современных стандартов шифрования (TLS 1.2 или выше).

* **Whitelisting:** Для on-premise инсталляций, которые часто находятся за корпоративными фаерволами, стандартной практикой является настройка белых списков IP-адресов. Партнеры (например, Notarize) предоставляют диапазоны своих IP-адресов, которые должны быть разрешены на периметре сети титульной компании.6

## **4. Спецификация REST API: Эндпоинты и Структуры Данных**

Хотя публичная спецификация Swagger отсутствует, анализ интеграционных сценариев позволяет реконструировать основные группы эндпоинтов и логику их работы.

### **4.1. Ресурс: Файлы (Files)**

Файл (File) в терминологии Resware — это центральный агрегат данных, представляющий собой отдельную сделку (Order/Transaction). Большинство операций API происходит в контексте конкретного файла.

#### **4.1.1. Поиск Файла (File Search)**

Прежде чем совершить действие с файлом, внешняя система должна узнать его внутренний идентификатор (FileID). Внешние системы обычно оперируют номером файла (File Number), который является человеко-читаемым, но не уникальным в глобальном смысле (может повторяться в разных офисах).

* **Эндпоинт:** POST /Api/files/search 3  
* **Методология:** Использование POST для поиска (вместо GET) является распространенным паттерном в энтерпрайз-системах, когда параметры поиска могут быть сложными или чувствительными.  
* **Входные данные:** JSON-объект, содержащий FileNumber и, возможно, OfficeID.  
* **Ответ:** JSON-объект, содержащий FileID (внутренний GUID или Integer), который используется во всех последующих запросах.

### **4.2. Ресурс: Документы (Documents)**

Управление документами — одна из самых активных функций API, используемая для загрузки результатов работы внешних систем (подписанные договоры, отчеты о поиске, справки).

#### **4.2.1. Загрузка Документа**

* **Эндпоинт:** POST /Api/files/{FileID}/documents 3  
* Механизм передачи:  
  Загрузка обычно осуществляется с использованием multipart/form-data или JSON с Base64-энкодированным контентом (в зависимости от конкретной реализации клиента и версии API).  
* Метаданные:  
  Критически важным аспектом является передача правильного DocumentTypeID. Resware использует строгую классификацию документов. Если загрузить документ с неверным ID типа, он может не отобразиться в нужном месте интерфейса или не запустить автоматический триггер.  
  * *Пример:* При интеграции с Notarize документ загружается с типом "Notarize Executed Document". Имя файла часто модифицируется для включения имени подписанта (например, DocName_SignerName_notarized.pdf).3

#### **4.2.2. Получение Документов**

API позволяет выгружать документы для обработки. Это требует предварительного мэппинга типов документов. Например, "Closing Disclosure", созданный в Resware, должен быть передан в RON-платформу. API позволяет получить бинарный контент файла по его ID.

### **4.3. Ресурс: Заметки (Notes) и Коммуникация**

Заметки в Resware — это не просто текстовые комментарии. Это основной механизм **событийно-ориентированной архитектуры** (Event-Driven Architecture) платформы.

#### **4.3.1. Создание Заметки**

* **Эндпоинт:** POST /Api/files/{FileID}/notes 3  
* Бизнес-логика:  
  Интеграторы используют этот эндпоинт для сигнализации о завершении внешнего процесса. Например, после того как все документы загружены, партнер отправляет заметку с текстом "Notarization Complete".  
* Автоматизация (Triggers & Actions):  
  Внутри Resware настраиваются "Действия" (Actions), которые слушают появление заметок с определенным текстом или от определенного партнера. При поступлении такой заметки система может автоматически:  
  * Сменить статус дела.  
  * Отправить уведомление риелтору.  
  * Сгенерировать финальный полис.  
    Это превращает REST API в мощный инструмент оркестрации процессов без необходимости прямого вызова внутренних процедур workflow.

### **4.4. Работа с Расширенными Данными (Custom Fields)**

Стандартной схемы данных часто недостаточно для специфических нужд. Resware предоставляет поддержку кастомных полей, доступ к которым также возможен через API.

* **Сценарий использования:** При интеграции с Proof (Notarize) для поддержки подписантов-юридических лиц (Entities) используются кастомные поля для хранения имен и титулов уполномоченных представителей (signatories).13  
* Реализация:  
  Необходимо создать соответствующие поля в административной панели Resware (например, Entity buyer signatory 1 first name). API позволяет читать и обновлять значения этих полей, привязывая их к конкретному файлу. Это требует права Web Services: Get Custom Fields.

## **5. Протоколы Наследия: XML и Resware-to-Resware**

Несмотря на фокус на REST, значительная часть функциональности Resware опирается на XML-обмен, который остается безальтернативным для ряда задач.

### **5.1. Протокол XMLORDER**

Для создания заказов (Order Placement) стандарт REST часто уступает место протоколу XMLORDER.

* **Точка входа:** Обычно это URL вида https://<resware-domain>/eService/?mode=XMLORDER....4  
* **Структура:** Данные передаются в виде XML-пакета, который должен строго соответствовать XSD-схеме Resware. Сюда входят данные о собственности, сторонах сделки, типе транзакции и заказываемых услугах.  
* **Синхронность:** В отличие от многих REST-методов, XMLORDER может работать в асинхронном режиме, где подтверждение приема заказа приходит мгновенно, а сама обработка и создание файла происходят в фоновом режиме.

### **5.2. Resware-to-Resware (R2R)**

Это специализированный протокол для связи двух независимых инстансов Resware (например, Титульное Агентство <-> Андеррайтер).

* **Особенности:** R2R позволяет "прокидывать" данные и документы между системами, сохраняя структуру.  
* **Конфигурация:** Настраивается через Partner Mappings. Важно, что здесь используются Remote ID — идентификаторы типов транзакций, продуктов и документов в *удаленной* системе, которые должны быть сопоставлены с локальными ID.4  
* **Применение в API:** Некоторые интеграторы (например, Notaroo) эмулируют поведение R2R-партнера, предоставляя API, совместимый со спецификациями R2R, для получения данных о заказах.6

## **6. Процесс Интеграции: Пошаговое Руководство**

На основе консолидированных данных из различных источников, процесс технической реализации интеграции можно разделить на несколько этапов.

### **6.1. Этап 1: Административная Подготовка**

До написания первой строки кода необходимо выполнить масштабную работу в консоли администратора Resware.

1. **Создание Партнера:** В разделе Admin/Partners создается запись для внешнего сервиса (например, "A.S.K. Services" или "Notarize"). Тип партнера (Searcher, Signing Service) определяет доступные функции.4  
2. **Настройка Сотрудника:** Создается "Partner Employee" с привязкой к созданному партнеру. Генерируется пароль, который будет использоваться для Basic Auth.  
3. **Экспорт ID (Seed Data):** Разработчику интеграции необходимы внутренние ID системы. Администратор должен выгрузить отчеты (Data Dumps) из разделов Document Types, Transaction Types, Lien Types. Без знания того, что DocumentTypeID=145 соответствует "Deed", интеграция невозможна.4

### **6.2. Этап 2: Конфигурация Мэппинга (Partner Mappings)**

Это "мозг" интеграции. В разделе ResWare to ResWare Partner Mappings задаются правила трансляции данных.

* **Service Type:** Выбирается тип взаимодействия (например, Order Placement или Note/Document).  
* **URL:** Указывается эндпоинт партнера, куда Resware будет отправлять данные (или откуда будет принимать).  
* **ID Mapping:** Устанавливается соответствие: "Когда я отправляю документ типа X, партнер должен видеть его как тип Y" (Remote ID).4

### **6.3. Этап 3: Разработка Клиента (Client Implementation)**

Разработчик создает приложение, которое:

1. Аутентифицируется через Base64-кредешиалы.  
2. Слушает входящие запросы от Resware (если используется XMLORDER push) или поллит Resware на предмет новых задач.  
3. Выполняет REST-запросы для обновления статусов.  
   * *Пример кода (псевдокод) для загрузки документа:*  
     POST /Api/files/12345/documents HTTP/1.1  
     Host: resware.example.com  
     Authorization: Basic <Base64_User_Pass>  
     Content-Type: multipart/form-data; boundary=---Boundary

     -----Boundary  
     Content-Disposition: form-data; name="document"; filename="SignedDeed.pdf"  
     Content-Type: application/pdf

     -----Boundary  
     Content-Disposition: form-data; name="DocumentTypeID"

     145  
     -----Boundary--

### **6.4. Этап 4: Настройка Автоматизации (Actions & Triggers)**

Для завершения цикла интеграции в Resware настраиваются триггеры.

* *Сценарий:* Когда через API поступает документ с типом "Search Package" от партнера "A.S.K.", автоматически создать задачу "Review Search" для сотрудника отдела Examination.4

## **7. Экосистема Партнеров и Примеры Использования**

Анализ существующих интеграций демонстрирует гибкость API.

### **7.1. Simplifile (E-Recording)**

Интеграция с Simplifile (часть ICE Mortgage Technology) позволяет отправлять документы на электронную регистрацию напрямую из Resware.

* **Механизм:** Использует комбинацию экспорта документов и обратного получения данных о записи (Recording Information).  
* **Особенности:** Требует уровня лицензии "Enterprise" для некоторых функций глубокой интеграции.14 Обеспечивает бесшовный статус-трекинг: как только документ записан в округе, информация об этом (Book/Page numbers) автоматически обновляется в Resware через API.15

### **7.2. Notarize / Proof (RON)**

Один из самых документированных примеров использования гибридного API.

* **Поток:**  
  1. Resware отправляет XML-заказ в Proof (создание транзакции).  
  2. Proof проводит сессию видео-подписания.  
  3. Proof использует REST API (/Api/files/...) для возврата подписанных PDF и аудиторского лога (Audit Trail) обратно в Resware.12  
* **Особенность:** Использование кастомных полей для поддержки сложных корпоративных структур подписантов.13

### **7.3. Shopware vs Resware: Важное Уточнение**

В процессе анализа технических источников можно столкнуться с путаницей между "Resware" и "Shopware" (популярная e-commerce платформа). Важно подчеркнуть, что упоминания эндпоинтов /store-api/ или манифестов manifest.xml для плагинов относятся к Shopware и не имеют отношения к Resware.16 Архитектура Resware строится на WCF/SOAP/REST стеке Microsoft, а не на PHP-стеке Shopware.

## **8. Проблемы, Решения и Операционная Стратегия**

### **8.1. Типичные Проблемы Интеграции**

1. **Проблемы с Аутентификацией WCF:** Самая частая проблема — попытка использовать Basic Auth на сервере, где эта опция не активирована на уровне IIS/WCF конфигурации. Решение: запрос в техподдержку.6  
2. **Несоответствие ID:** Использование ID из тестовой среды в продакшене. В Resware ID типов документов и продуктов могут различаться между инстансами. Решение: всегда делать свежий экспорт (Data Dump) ID перед развертыванием.4  
3. **Ошибки Валидации XML:** При использовании XMLORDER малейшее отклонение от XSD-схемы приводит к отказу. Решение: использование валидаторов и строгое следование документации партнера.

### **8.2. Стратегия Мониторинга**

Для надежной работы интеграции необходимо внедрять логирование на стороне клиента API. Поскольку Resware не предоставляет внешним системам прямого доступа к своим логам, клиент должен записывать:

* Все исходящие запросы (с маскированием PII).  
* HTTP-коды ответов.  
* Тела ответов ошибок (Resware часто возвращает детали ошибки в JSON/XML теле).

### **8.3. Влияние Обновлений (Resware 10)**

Переход на Resware 10 требует ревизии интеграций. Хотя обратная совместимость декларируется, изменения в инфраструктуре безопасности или производительности могут повлиять на тайм-ауты и поведение коннекторов. Рекомендуется проводить регрессионное тестирование всех API-потоков в песочнице перед обновлением продакшена.10

## **9. Заключение**

Resware REST API является критически важным инструментом для современной титульной компании, позволяющим преодолеть ограничения монолитной архитектуры и построить гибкую, автоматизированную среду. Несмотря на отсутствие открытой публичной документации и наличие технологического наследия в виде XML/SOAP, API предоставляет достаточный функционал для реализации сложных интеграционных сценариев.

Успех интеграции зависит не столько от сложности кода, сколько от глубокого понимания административной структуры Resware (Roles, Mappings, Action Groups) и тесного взаимодействия с вендором. С выходом Resware 10 и развитием экосистемы Qualia, роль стандартизированных API будет только расти, открывая новые возможности для инноваций в сфере недвижимости.

**Таблица 2. Сводная характеристика методов интеграции Resware**

| Характеристика | REST API | XMLORDER (eService) | Resware-to-Resware |
| :---- | :---- | :---- | :---- |
| **Основное назначение** | Работа с документами, заметками, статусами | Создание заказов, массовый импорт | Связь между системами Resware |
| **Формат данных** | JSON (преимущественно), Multipart | XML | Проприетарный XML / Binary |
| **Транспорт** | HTTPS (POST, GET) | HTTPS (POST) | WCF / HTTPS |
| **Сложность внедрения** | Средняя (требует знания HTTP/JSON) | Высокая (сложные XML схемы) | Низкая (настройка через UI) |
| **Гибкость** | Высокая (управление ресурсами) | Низкая (строгий контракт) | Средняя (ограничено функционалом R2R) |

## **Аннотация**

В современной индустрии титульного страхования и сопровождения сделок с недвижимостью (Title & Escrow) программное обеспечение является не просто инструментом учета, а центральной нервной системой предприятия. Платформа Resware, разработанная Adeptive Software (ныне являющаяся ключевым активом в портфеле Qualia), представляет собой стандарт де-факто для автоматизации сложных транзакционных рабочих процессов. Ключевым механизмом, обеспечивающим интероперабельность Resware с внешним миром — от платформ удаленного нотариата до систем андеррайтинга — является технология **Action Events (События Действий)**.

Данный отчет представляет собой фундаментальное исследование архитектуры, конфигурации и стратегического применения Action Events. На основе анализа технической документации, протоколов интеграции партнеров (Snapdocs, Proof, A.S.K. Services, First American) и административных руководств, мы деконструируем механизм работы событий. Мы рассмотрим, как Action Events трансформируют статические списки задач в динамические, событийно-ориентированные потоки данных, обеспечивая бесшовную синхронизацию статусов, документов и финансовых данных через защищенные XML-шлюзы. Отчет предназначен для системных архитекторов, администраторов Resware и технических директоров титульных агентств, стремящихся к максимальной операционной эффективности.

## **Часть I: Концептуальные Основы и Архитектура Рабочих Процессов Resware**

### **1.1. Эволюция Систем Производства Титулов**

Индустрия недвижимости исторически характеризовалась фрагментарностью. Титульные агенты, кредиторы, нотариусы и государственные регистраторы работали в изолированных информационных системах, что приводило к дублированию данных и задержкам. Resware была спроектирована как решение этой проблемы, предлагая не просто базу данных, а "движок рабочих процессов" (workflow engine).

В экосистеме Qualia, куда теперь входит Resware, продукты разделены по функциональным нишам: **Atlas** для корпоративных систем, **Connect** для взаимодействия с потребителями и **Resware** как настраиваемое ядро производства титулов.1 Основная ценность Resware заключается в возможности создания кастомных решений для автоматизации.1 В этом контексте "автоматизация" означает способность системы самостоятельно реагировать на изменения состояний без участия человека.

### **1.2. Иерархия Управления Задачами: От Action Lists к Глобальным Событиям**

Чтобы понять природу Action Events, необходимо рассмотреть структуру управления задачами, в которую они интегрированы. Resware оперирует понятием **Action Lists (Списки Действий)** — предопределенными последовательностями шагов, необходимых для закрытия файла.2

#### **Динамическая Природа Action Lists**

Списки действий в Resware не являются линейными чеклистами. Это сложные графы зависимостей, которые могут изменяться в реальном времени.

* **Контекстуальность:** Состав действий зависит от типа транзакции (рефинансирование, покупка, продажа), штата, округа и выбранного андеррайтера.1  
* **Триггеры (Triggers):** Действия могут запускаться автоматически. Система поддерживает триггеры для "автоматического заказа поисков", "автоматического добавления комиссий", "запуска групп действий для других пользователей".1  
* **Аффекты (Affects):** Завершение одного действия может вызывать каскад последствий (Affects), включая изменение статуса файла, отправку уведомлений или генерацию документов.

#### **Дилобальные Действия (Global Actions)**

Существует критическое различие между обычными действиями и **Глобальными Действиями (Global Actions)**.

* **Обычные Действия:** Привязаны к конкретному списку и жизненному циклу файла.  
* **Глобальные Действия:** Доступны из любого контекста и часто используются для общесистемных функций, таких как интеграция с внешними сервисами. Например, в интеграции с First American используется глобальное действие для отправки события "Post-Pricing" (после ценообразования), которое передает андеррайтеру финальные суммы полисов.3

### **1.3. Определение Action Event**

В терминологии Resware, **Action Event (Событие Действия)** — это специализированный механизм интерфейса, предназначенный для обмена дискретными сигналами о статусе задач между ядром Resware и внешней системой.

Если Action List управляет внутренней логикой ("Что делать сотруднику?"), то Action Event управляет внешней синхронизацией ("Что произошло во внешнем мире?" или "О чем нужно сообщить внешнему миру?"). Это мост, по которому протекают данные в формате XML, связывая локальный сервер Resware (On-premises) с облачными платформами партнеров.

## **Часть II: Техническая Архитектура Интеграции**

### **2.1. Протокол XML: Лингва Франка Resware**

В основе всех Action Events лежит обмен XML-сообщениями. Resware не использует современные JSON REST API в их классическом понимании для ядра интеграции событий, полагаясь на проверенную структуру XML, которая обеспечивает строгую типизацию данных.

#### **ResWare-to-ResWare (R2R) Интерфейс**

Хотя название подразумевает связь двух систем Resware, этот интерфейс (ResWare-to-ResWare) стал стандартом для подключения *любого* внешнего партнера, который эмулирует поведение Resware. Вендоры, такие как Snapdocs, Notaroo или A.S.K. Services, разрабатывают свои API так, чтобы они "выглядели" для Resware как другой экземпляр Resware.4

#### **Типы Сервисов (Service Types)**

При настройке Action Events администратор должен выбрать правильный **Service Type** в маппинге партнеров. Анализ документации выявляет четыре критических типа сервисов 4:

| Тип Сервиса (Service Type) | Назначение | Применение в Action Events |
| :---- | :---- | :---- |
| **Order Placement** | Первичное создание заказа. Передает данные о недвижимости, заемщике и номер файла. | Используется как триггер для начала работы партнера (например, заказ поиска в A.S.K.). |
| **Signing** | Управление процессом подписания. Передает дату, время и место встречи. | Критичен для интеграции с платформами RON (Proof, Snapdocs). |
| **Note/Document** | Передача неструктурированных данных (заметок) и бинарных файлов (PDF). | Используется для возврата подписанных документов в Resware. |
| **Action Events** | Передача кодов статусов. | Основной канал для синхронизации прогресса (например, "Нотариус назначен"). |

### **2.2. Механизм Кодирования Событий (Action Codes)**

Одной из самых уникальных особенностей архитектуры Resware является использование произвольных числовых кодов для Action Events. Система не навязывает жесткий словарь событий; вместо этого она предоставляет механизм маппинга.

* **Принцип работы:** Администратор создает событие с понятным именем (например, notary_assigned) и присваивает ему произвольный код (например, 44).  
* **Синхронизация:** Партнер (например, Snapdocs) должен быть настроен на отправку именно кода 44 при наступлении этого события. Если партнер отправит код 45, Resware не распознает событие, даже если имя совпадает.2  
* **Гибкость:** Этот подход позволяет интегрировать Resware с любыми системами, даже если их внутренние коды статусов отличаются — достаточно настроить таблицу соответствия (translation table) на уровне партнера.

### **2.3. Вкладка XML: Центр Диагностики**

Все входящие и исходящие сообщения Action Events логируются в специальной вкладке **XML** внутри файла заказа. Это критический инструмент для администраторов.

* **Прозрачность:** Здесь отображаются сырые (raw) запросы и ответы.  
* **Отладка:** В случае сбоя интеграции именно здесь появляются сообщения об ошибках, такие как "Invalid credentials" или ошибки несоответствия типов подписантов.6

## **Часть III: Глубокая Настройка и Конфигурация**

Настройка Action Events — это многоступенчатый процесс, требующий высокой точности. Ошибка на любом этапе приведет к разрыву цепочки автоматизации. Ниже приведен синтезированный алгоритм настройки, основанный на руководствах Snapdocs 2, First American 7 и Stewart/A.S.K..5

### **Этап 1: Подготовка Инфраструктуры (General Setup)**

Прежде чем создавать события, необходимо разрешить обмен данными на уровне офиса.

1. **Включение Веб-сервисов:** В разделе Admin/General Setup/Offices необходимо выбрать офис и во вкладке "Web Services" активировать опцию Enable ResWare to ResWare Order Submission.5 Без этого флага система будет блокировать любые входящие XML-запросы по соображениям безопасности.  
2. **Настройка Продуктов:** В Admin/Action Lists/Product Types для каждого продукта, который будет участвовать в интеграции (например, "Refinance"), необходимо добавить пакет XML ResWare.5 Это говорит системе, что файлы этого типа *могут* обрабатывать XML-сообщения.

### **Этап 2: Создание Партнера и Маппинг (Partner Configuration)**

Это ядро интеграции. Здесь устанавливается доверительное отношение между Resware и внешней системой.

1. **Создание Партнера:** В Admin/Partners создается запись для вендора (например, "Snapdocs" или "A.S.K. Services").  
2. **Настройка Маппинга:** Переход в Admin > Partner-Related > ResWare to Resware Partner Mappings.  
   * **Выбор Партнера:** Выбираем созданного вендора.  
   * **URL и Аутентификация:** Ввод URL целевого сервиса. В случае интеграции с Notaroo, URL может содержать ключ подписки: https://.../order?subscription-key={Key}.4 Для A.S.K. Services требуются Client ID и Office ID.5  
   * **Выбор Типа Сервиса:** Здесь администратор определяет канал. Для передачи статусов выбирается подкатегория **Action Events**. Для отправки заказов — **Order Placement**.

### **Этап 3: Определение Событий (Defining Action Events)**

1. Переход в Admin > Action Lists > Action Events.  
2. Добавление семантических имен событий. Рекомендуется использовать стандартизированные названия для удобства чтения:  
   * notary_assigned  
   * appointment_confirmed  
   * notary_downloads_docs  
   * completion_status_did_not_sign (критическое событие для обработки срывов сделок).2

### **Этап 4: Связывание Кодов (Code Assignment)**

Это этап "перевода" человекочитаемых имен в машинные коды.

1. Возвращаемся в ResWare to Resware Partner Mappings.  
2. Выбираем созданного партнера и категорию Action Events.  
3. Нажимаем Add и для каждого события из Этапа 3 назначаем числовой код (например, 44 для notary_assigned).  
   * *Важно:* Эти коды должны быть предоставлены партнером по интеграции. Нельзя придумывать их самостоятельно, если партнер ожидает жестко заданные значения.2

### **Этап 5: Интеграция в Workflow (Action Affects)**

События настроены, но они еще ничего не делают. Необходимо привязать их к Action List.

1. Переход в Admin > Action Lists > Action Groups.  
2. Выбор действия, которое должно реагировать на событие (например, "Assign Notary").  
3. Настройка **Affects** (Последствий):  
   * *Входящее событие:* Настраивается так, чтобы получение кода 44 автоматически помечало действие "Assign Notary" как выполненное (Complete Task).  
   * *Исходящее событие:* В Edit Affects -> Marking Done Sends XML выбирается ResWare: Send Order для инициации отправки заказа партнеру при завершении действия сотрудником.5

## **Часть IV: Экосистема Партнеров и Кейсы Интеграции**

Анализ исследовательских материалов позволяет выделить несколько архетипов интеграции Action Events, каждый из которых решает специфические бизнес-задачи.

### **4.1. Архетип "Синхронизация Статусов": Snapdocs**

Snapdocs специализируется на управлении закрытием сделок. Интеграция с Resware фокусируется на прозрачности процесса для всех участников.

* **Особенность реализации:** Snapdocs предлагает три уровня интеграции 8:  
  1. *Outlook Integration:* Парсинг email-уведомлений. Resware создает заметку (Note), когда приходит письмо с номером файла в теме. Это базовый уровень без использования Action Events.  
  2. *Client Integration:* Для сервисов подписания, работающих с клиентами на Resware.  
  3. *XML Integration (Action Events):* Полноценный обмен кодами.  
* **Специфика VendorPay:** Snapdocs позволяет интегрировать статусы оплаты нотариусов. Событие payment_processed может автоматически обновлять бухгалтерский модуль Resware, закрывая обязательства перед вендором.8

### **4.2. Архетип "Документарный Оборот": Proof (ранее Notarize)**

Интеграция с платформой удаленного онлайн-нотариата (RON) Proof является наиболее сложной, так как требует передачи не только статусов, но и документов с точным позиционированием подписей.

#### **Технология White Text Tagging**

Для того чтобы Action Event "Send to Proof" сработал корректно, документы должны быть подготовлены специфическим образом. Resware не просто отправляет PDF; она генерирует документы, содержащие **"белый текст" (White Text Tags)** — невидимые символы, которые служат якорями для платформы Proof.9

* *Механизм:* При генерации документа Resware вставляет теги (например, {{Sig_1}}) белым шрифтом на белом фоне. Proof сканирует входящий XML-пакет и документ, находит эти теги и размещает интерактивные поля подписи именно в этих координатах. Без этой технологии автоматизация невозможна — документ потребует ручной разметки ("drag-and-drop") на стороне Proof.

#### **Поддержка Юридических Лиц (Entity Support)**

Интеграция Proof поддерживает сложные сценарии, когда продавцом выступает юридическое лицо (траст, корпорация).

* **Кастомные Поля (Custom Fields):** Стандартных полей Resware недостаточно. Администраторы должны создать специальные кастомные поля: Entity buyer signatory 1 email, Entity buyer signatory 1 title и т.д..10  
* **Логика:** Action Event считывает данные из этих кастомных полей и передает их в XML-структуре, позволяя Proof корректно идентифицировать подписанта, действующего от имени компании.

### **4.3. Архетип "Автоматизация Заказа": A.S.K. Services**

A.S.K. Services (поставщик поисковых услуг) использует Action Events для полной автоматизации цикла заказа.

* **Триггер:** Глобальное действие A.S.K.: Request Search.  
* **Условия:** Действие настраивается как "Dynamic" и требует наличия документа (например, контракта купли-продажи) перед выполнением.5 Это предотвращает отправку пустых заказов.  
* **Двунаправленность:**  
  * *Исходящий:* ResWare: Send Order (тип Service Type: Order Placement).  
  * *Входящий:* Результаты поиска возвращаются через сервис Note/Document. Здесь критично настроить маппинг типов документов: внешний Remote ID документа должен соответствовать внутреннему Document Type в Resware (например, "Search Abstract").5

### **4.4. Архетип "Финансовый Андеррайтинг": First American**

Интеграция с First American (FA) демонстрирует использование Action Events для финансовых операций.

* **Post-Pricing Events:** После того как в Resware рассчитаны премии (Rates & Fees), инициируется специальное событие "Post-Pricing". Оно отправляет данные о полисах и суммах обратно в систему FA для валидации.3  
* **Обработка Расхождений:** Если суммы в Resware отличаются от расчетных сумм FA, система генерирует предупреждение (popup) и добавляет заметку в файл. Это пример того, как Action Event работает как механизм аудита.  
* **Policy Upload:** Отдельное действие настраивается для автоматической загрузки финального пакета полиса в систему FA через вкладку XML Clients > Policy Upload.7

### **4.5. Архетип "Безопасная Коммуникация": RMail**

Интеграция с RMail (RPost) показывает, как Action Events могут управлять безопасностью email-переписки.

* **Файловые Типы:** Для работы функции "Proof of Delivery" (Доказательство доставки) требуется, чтобы в Resware был активирован тип файла .htm.11  
* **Автоматизация:** Шаблоны писем (Email Templates) настраиваются на автоматическое использование функций Track & Prove или Encrypt. Action Event здесь — это сам факт отправки письма, который триггерит создание защищенного конверта и последующее прикрепление квитанции о доставке обратно к файлу.11

## **Часть V: Продвинутая XML-Механика и Безопасность**

### **5.1. Структура XML Payload и Валидация**

XML-схемы Resware (XSD) являются проприетарными, однако анализ ошибок и документации позволяет реконструировать их ключевые элементы.

* **Идентификаторы:** Ключевыми полями являются FileNumber, PartnerID и TransactionType.  
* **Вложенность:** Сообщения часто имеют сложную вложенную структуру. Например, при передаче данных о клиентах используется иерархия <Customers><Customer><Address>...</Address></Customer></Customers>.12  
* **Схемы и Валидация:** Важно отметить, что, как и во многих корпоративных интеграциях (пример IBM Maximo 13), строгая валидация по XSD-схеме может быть отключена для повышения гибкости, полагаясь вместо этого на валидацию бизнес-правил на уровне приложения. Однако Resware строго проверяет пространство имен (namespace) и структуру заголовков.

### **5.2. Обработка Ошибок и Логирование**

Action Events не всегда проходят успешно. Resware предоставляет механизмы обратной связи через XML-ответы.

**Таблица 3: Типичные Ошибки Интеграции и Методы Устранения**

| Сообщение об Ошибке (XML Log) | Вероятная Причина | Метод Устранения |
| :---- | :---- | :---- |
| **"Invalid credentials"** | Неверный URL, Логин/Пароль или ключ подписки в Partner Mapping.6 | Проверить URL в маппинге. Убедиться, что параметры (например, subscription-key) не обрезаны. |
| **"Signing party... does not match transaction type"** | Попытка отправить продавца в транзакцию типа "Покупка" (Buyer side only). | Проверить соответствие Signing Type в Resware и Transaction Type на платформе партнера (Proof). |
| **Код события не распознан** | Партнер прислал код, который не добавлен в маппинг Action Events. | Добавить новый код в Admin > Partner-Related > Mappings. |
| **Отсутствуют документы** | Глобальное действие запущено до генерации обязательных документов. | Проверить условие Completing Requires Documents в настройках Global Action.5 |

### **5.3. Безопасность Данных**

Передача данных через Action Events затрагивает чувствительную информацию (NPPI — Non-Public Personal Information).

* **Шифрование:** Все XML-транзакции должны проходить через HTTPS.  
* **Изоляция Офисов:** Настройка Enable ResWare to ResWare Order Submission производится на уровне конкретного офиса, что позволяет сегментировать доступ в многофилиальных организациях.5

## **Часть VI: Операционное Влияние и Будущее Технологии**

### **6.1. Влияние на Бизнес-Показатели (KPI)**

Внедрение Action Events напрямую влияет на ключевые метрики эффективности титульного агентства.

* **Сокращение Turnaround Time:** Автоматический заказ поисков и мгновенное получение результатов через XML устраняет часы простоя, ожидая ручной обработки email.  
* **Масштабируемость:** Использование стандартизированных кодов позволяет легко менять вендоров (например, переход от одного нотариального сервиса к другому), меняя только URL и коды маппинга, без перестройки внутренних процессов.  
* **Снижение Ошибок:** Исключение ручного ввода данных (re-keying) из писем в систему снижает риск опечаток в суммах полисов и именах клиентов.

### **6.2. Qualia и Будущее Автоматизации**

С приобретением Adeptive компанией Qualia, Resware становится частью более широкой экосистемы.

* **Облачная Миграция:** Традиционно Resware была on-premise решением (установленным на серверах клиента). Qualia активно развивает облачные интеграции (Qualia Atlas), где Action Events могут эволюционировать в более современные API (REST/GraphQL).  
* **Agentic AI:** В документации Qualia упоминается "Agentic AI System" (Qualia Clear).1 В будущем Action Events могут стать триггерами не для жестких скриптов, а для AI-агентов, способных принимать сложные решения (например, самостоятельно разрешать расхождения в суммах полисов, основываясь на историческом контексте, а не просто выдавать ошибку).

## **Заключение**

Action Events в Resware — это фундамент, на котором строится современная автоматизированная фабрика титулов. Несмотря на кажущуюся простоту концепции (обмен кодами и XML), глубина их настройки и влияние на бизнес-процессы колоссальны. От корректности "белых тегов" в PDF до точности маппинга кодов статусов — каждая деталь имеет значение. Для профессионалов индустрии владение этим инструментарием является обязательным условием для построения конкурентоспособного, масштабируемого и защищенного бизнеса в цифровую эпоху.

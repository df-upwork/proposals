1) «the new sales tax setup makes no sense»
Система имеет четкую логику, направленную на соблюдение сложного налогового законодательства.
Калифорния применяет гибридную модель (modified origin sourcing) для внутриштатных продаж.
Налоги штата, округа и города (State, county, city taxes) рассчитываются по месту нахождения продавца (origin-based).
Районные налоги (district taxes) рассчитываются по месту доставки (destination-based).
AST автоматизирует учет этих сложных правил и тысяч юрисдикций в Bay Area, что практически невозможно делать вручную.
2) «seems that sales tax rate needs to be added manually» 
Это не так.
Цель AST — именно устранить необходимость ручного отслеживания и ввода ставок. 
Intuit автоматически обновляет ставки при изменении законодательства.
3) «The customer section has a box checked for “sales tax” but there is nothing further than that»
Поскольку ставка зависит от точного адреса доставки в конкретной транзакции, который может меняться, хранение статической ставки в профиле клиента некорректно.
Поэтому AST намеренно не хранит ставку в профиле клиента. 
Профиль определяет только статус налогообложения (Taxable/Exempt). 
Intuit удалил возможность установки налогового кода по умолчанию при активации AST.
4) «I need to have clients sales tax rates setup and flowing to 3rd party tool»
4.1) Система QBO Automated Sales Tax (AST) рассчитывает налог динамически.
Стандартный QuickBooks Online Accounting REST API (V3) не позволяет стороннему приложению узнать рассчитанную AST ставку до того, как транзакция будет создана и сохранена в QBO.
4.2) Существует два основных подхода к решению этой проблемы интеграции.
Я излагаю их в пунктах 5 и 6 ниже.
Реализуемость любого из них зависит от технических возможностей используемого стороннего инструмента (3rd party tool) и его способности к интеграции с внешними API или модификации логики синхронизации с QBO.
5) Подход 1: Внедрение специализированного налогового сервиса (e.g., Avalara, TaxCloud, TaxJar).
Сторонний инструмент интегрируется с API этого сервиса для получения точной ставки в реальном времени.
6) Подход 2: Интеграция стороннего инструмента с Intuit Sales Tax API (GraphQL) для получения ставки, рассчитанной AST, в реальном времени.
7) В обоих подходах при последующей синхронизации транзакции с QBO через Accounting REST API (V3) необходимо использовать механизм переопределения налога (Sales Tax Override).
Это требует не только передачи итоговой суммы через поле `TxnTaxDetail.TotalTax`.
Для надежного переопределения также требуется ссылка на пользовательский налоговый код (Custom Sales Tax Code, тип `USER_DEFINED`) через поле `TxnTaxDetail.TxnTaxCodeRef` и использование актуальной версии API (`minorversion`).
8) Подход 1 обладает следующими преимуществами по сравнению с Подходом 2:
8.1) Повышенная точность и комплаенс.
Специализированные сервисы используют продвинутые методы валидации адресов и геолокации (address-level accuracy) для учета всех местных и районных налогов (district taxes).
Это обеспечивает максимальную корректность расчета в сложных юрисдикциях (Bay Area).
8.2) Предоставляет собственный мощный API.
Сторонний инструмент может быть интегрирован напрямую с этим ПО для получения ставок в реальном времени, минуя ограничения QBO Accounting REST API.
8.3) Автоматизация комплаенса.
Автоматическое обновление ставок, корректная отчетность для CDTFA и автоматическая подача деклараций (AutoFile). 
8.4) Стабильное, масштабируемое решение, не зависящее от ограничений или изменений в API Intuit.

1) The online version of Business Central (`BC`) не поддерживает шрифт «Century Gothic» (`CG`): https://learn.microsoft.com/en-us/dynamics365/business-central/ui-fonts#document-fonts

3) Способ №1: шрифт можно встроить в макет отчёта в формате Word.
3.1) Пошаговая инструкция.
3.1.1) В `BC` необходимо перейти на страницу «Report Layout Selection» (`RLS`). 
Там нужно найти требуемый отчет и экспортировать его текущий макет типа Word в файл `.docx` (`F`).
3.1.2) Открыть `F` в Microsoft Word (`W`). 
3.1.3) Применить `CG` к необходимым текстовым полям и элементам управления содержимым.
3.1.4)  В `W` перейти в меню Файл > Параметры > Сохранение. 
В открывшемся окне следует активировать опцию «Embed fonts in the file». 
Для оптимизации размера файла также настоятельно рекомендуется активировать опцию «Do not embed common system fonts».
3.1.5) Сохранить `F`.
3.1.6) В `BC` на странице «Custom Report Layouts» создать новый макет (`L`) и импортировать в него `F`.
3.1.7) Назначить `L` в качестве макета по умолчанию для данного отчета на странице `RLS`.
3.2) Техническое обоснование работоспособности
3.2.1) Механизм, лежащий в основе этого метода, связан с процессом генерации PDF-файлов на сервере `BC`.
3.2.2) Когда система генерирует PDF из макета Word, она использует серверный компонент (это библиотека Aspose).
3.2.3) Если шрифт не встроен в `.docx` файл, этот компонент считывает название шрифта, пытается найти его в списке установленных на сервере шрифтов, не находит и заменяет его на шрифт по умолчанию (например, Arial), что и приводит к вашей проблеме.
3.2.4) При активации опции встраивания полная информация о глифах и метриках шрифта сохраняется непосредственно внутри `.docx` файла. 
Когда этот файл обрабатывается серверным компонентом, он может извлечь данные шрифта из самого документа и использовать их для корректного рендеринга текста в PDF, не требуя при этом, чтобы шрифт был установлен в операционной системе сервера.

Для реализации Способа №1 важно, чтобы свойство «Font Embeddabilty» шрифта имело значение «Installable».

4) Способ №2: использование стороннего расширения для расширенной отчетности (ForNAV)
https://appsource.microsoft.com/en-us/product/dynamics-365-business-central/pubid.fornav%7Caid.customizable_report_pack%7Cpappid.83326d6d-11f8-49fd-981a-6f266a7c8d81
ForNAV — это комплексное решение для отчетности в `BC`, которое включает в себя собственный дизайнер отчетов, конвертер для стандартных отчетов и расширенный движок рендеринга. 
Ключевой особенностью ForNAV является его встроенная и официально поддерживаемая возможность работы с пользовательскими шрифтами.
В отличие от нативного обходного пути, встраивание шрифтов в ForNAV является основной, документированной функцией. 
Дизайнер отчетов ForNAV (ForNAV Designer) автоматически определяет, что в макете используется пользовательский шрифт, установленный на компьютере разработчика, и включает (встраивает) его данные непосредственно в файл макета отчета при сохранении.   
Собственный сервис рендеринга ForNAV, который обрабатывает эти макеты в среде `BC`, специально разработан для корректной работы с такими встроенными шрифтами. 
Это обеспечивает стабильный и предсказуемый результат, полностью устраняя неопределенность и риски, связанные с нативным компонентом рендеринга Microsoft.
5) Способ №3: Разработка внешнего микросервиса (например, Azure Function)
`BC` отправляет данные отчета (JSON/XML) в этот сервис. 
Сервис, используя сторонние библиотеки генерации PDF (например, PDFSharp, IronPDF), которые поддерживают полное внедрение шрифтов, генерирует документ с `CG` и возвращает его в BC.
Этот способ гарантирует использование любого шрифта и сложного форматирования, обходя все ограничения `BC`.
Генерируется высококачественный PDF с внедренным текстом (с возможностью поиска и копирования).

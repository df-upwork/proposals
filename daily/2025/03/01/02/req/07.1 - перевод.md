~~~
Наверняка источником вашей проблемы является изменении схемы данных MySQL, например:
1) Если в MySQL в таблице появилось новое поле, а в Power Query оно не добавлено в список выбираемых полей, то при обновлении отчёта значения этого поля не попадут в модель. 
2) Если в MySQL поле таблицы было переименовано или удалёно, а в Query Editor (M-коде) Power BI по-прежнему есть ссылка на прежнее имя, весь процесс обновления отчёта может прерываться, и новые данные не будут загружены.
3) Иногда Power Query тихо пропускает шаги, где не может найти удалённый столбец, в зависимости от настроек. 
4) Возможно, после изменения структуры таблиц в MySQL некоторые столбцы получили другие типы данных.
Например, если в MySQL поле, выступающее ключом, хранится как `VARCHAR`, а в Power BI оно импортируется как числовое (или наоборот), при сопоставлении значений (фильтрации, связях) часть новых строк может игнорироваться.  
5) Возможно, новый столбец в MySQL является ключевым для связи между таблицами, но в модели Power BI связи по-прежнему выстроены по старому полю. 
В этом случае а диспетчере связей (Manage Relationships) в Power BI связь может стать неактивной или оказаться помеченной ошибкой.  
6) Если Power BI воспринимает какое-то поле как ключ, но в реальности в MySQL теперь это поле содержит дубли, связь в Power BI рушится или становится неоднозначной.  
7) Несогласованность реляционных данных между связаными таблицами (изоляция данных): например, вставили новую строку в факт «Продажи», но в справочнике «Товары» отсутствует соответствующая запись о товаре или ключ (ProductID).  
Power BI, при сборке модели и связей, не находит нужной записи в измерении, а значит, фактическая строка остаётся одинокой и может не отображаться в отчёте (особенно если включены связи «один-ко-многим» или фильтрация «Both»).
8) Циклические (замкнутые) связи.  
Power BI не разрешает прямые циклы в отношениях, однако при наличии двусторонней фильтрации квазицикл может возникать через несколько промежуточных таблиц.  
В результате фильтры самопроизвольно пересекаются, и часть строк (включая новые) могут отбрасываться, когда вы выбираете что-то в другом срезе.
~~~

## 4. Правила перевода
### 4.1.
Переводи именно в той стилистике, как написано на русском языке.
Не делай перевод более вежливым, чем оригинал.

### 4.2.
Те предложения, которые сейчас полностью на английском — оставь без изменения.

### 4.3.
Не используй Markdown: только plain text.
При этом можно и нужно использовать то форматирование, которое уже есть в оригинале: его не убирай.

### 4.4.
Форматируй перевод в точности как оригинал. 
В частности:
*) каждый абзац должен содержать ровно одно предложение
*) между абзацами не должно оставаться пустых строк.
*) кавычки используй те же, что и в оригинале: «» и ``.

### 4.5.
Не используй сокращения типа «don't». Все подобные фразы пиши полностью: «do not».

### 4.6.
Не используй жаргон.
Вместо этого используй официальные термины.
#### 4.6.1.
В частности, фразы в кавычках используй только в том случае, когда они являются точными цитатами.
Не используй фразы в кавычках для применения жаргонных фраз.
Например, следующий фрагмент текста недопустим, потому что там используется жаргонная фраза «пролетел»: 
```
Например, код, который пушит данные о покупке, подключён асинхронно и загружается с небольшой задержкой, а триггер уже «пролетел».
```

### 4.7.
При обсуждении программного обеспечения используй точные официальные термины на английском языке: именно в том виде, как они указаны в официальной англоязычной документации к этому программному обеспечению.

### 4.8.
Не используй «you need» и другие подобные обращённые к клиенту фразы, перекладывающие действия на него.
Помни: я пишу клиенту или потенциальному клиенту.
Делать в любом случае буду я, а не клиент.
Вместо «you need» используй 2 альтернативы:
#### 4.8.1.
Нейтральные фразы типа «it is necessary».
#### 4.8.2.
Глаголы в неопределённой форме.
Например, во фрагменте ниже использованы подобные глаголы «set up», «create»:
```
1.2) Set up the transfer of login events from WordPress to Power BI using Fabric / OneLake.
1.2.1) Set up a «Data Pipeline» from the WordPress database table that stores login events (see point 1.1) to Fabric / OneLake.
1.2.2) Set up a connection from Power BI to Fabric / OneLake to pass login events.
1.3) Create the data model in Power BI.
```
Обрати внимание, в этом фрагменте не говорится, кто именно будет выполнять описанные действия: ответственность не перекладывается на клиента, в отличие от «you need».

### 4.9.
Никогда не переводи понятие «сайт» / «веб-сайт» как «site». 
Вместо этого используй форму «website»: это является более профессиональным.

### 4.10.
Никогда не переводи понятие «пункт нумерованного списка» как «item».
Всегда переводи это как «point».

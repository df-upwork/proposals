## 1.
Клиент хочет выполнить следующий проект:
### 1.1. Title
Microsoft Domain syncing
### 1.2. Description
Tenant 1 (acme.com): Hosted on Google, but also using Microsoft Teams for internal collaboration.
Tenant 2 (ugly.com): Fully hosted on Microsoft.
Goal: When scheduling Teams meetings in Tenant 1 via the Microsoft environment, you want to invite users from Tenant 2 and assign them as co-organizers.
Current Issue: Although Teams invites can be sent, the user accounts for Tenant 2 are not being created when scheduling the meetings, even though cross-domain chat works fine.
Additional Requirement: The Google-hosted domain for Tenant 1 must continue to function without interruption.

### 1.3. Tags
Microsoft Azure
Office 365
Microsoft Teams

## 2. В чём причина проблемы клиента?
### 2.1.
Я тебя спросил:
~~~

```
Although Teams invites can be sent, the user accounts for Tenant 2 are not being created when scheduling the meetings, even though cross-domain chat works fine.
```

### 2.2.
Ты ответил:
~~~
Судя по описанию, основная причина в том, что в **Tenant 1** (где создаётся встреча в Teams) **нет нужного механизма** или **не настроен процесс** для автоматического создания (или синхронизации) учётных записей пользователей из **Tenant 2**. 

Другими словами, когда организатор из Tenant 1 пытается назначить приглашённого пользователя из Tenant 2 сопроводителем (co-organizer), в **Azure AD** первого арендатора не существует «гостевой» записи для этого пользователя. Из‐за этого Microsoft Teams не может присвоить ему права, необходимые для роли организатора, ведь Teams «под капотом» полагается на Azure AD (или на сопоставление в «локальную» для арендатора директорию).

При этом **междоменный чат** (внешний федеративный чат) может работать, потому что для текстовой коммуникации требуется лишь включённая федерация между доменами/арендаторами (внешние чаты не создают полноценной учётной записи в чужом Azure AD). Но для роли «co-organizer» нужно именно существование пользователя (в виде guest или другого типа учётной записи) в Tenant 1. 

Таким образом, кратко:
1. **Чат работает**, потому что включена федерация между доменами в Teams, и для этого не нужно создавать учётную запись во внешнем арендаторе.  
2. **Назначение сопроводителем (co-organizer) не работает**, поскольку в Azure AD Tenant 1 не появляется запись пользователя из Tenant 2, следовательно, Teams не может назначить ему необходимые права в своей среде.
~~~

## 4. Правила ответа
### 4.1.
Инструкция должна быть на русском языке.
Исключением являются точные официальные термины: смотри пункт 4.2 ниже.

### 4.2.
При обсуждении программного обеспечения используй точные официальные термины на английском языке: именно в том виде, как они указаны в официальной англоязычной документации к этому программному обеспечению.

### 4.3.
Не используй выделение жирным (`**`) и курсив (`*`).

### 4.4.
Названия файлов заключай в backticks.
Например: `header.php`.

### 4.5.
Названия элементов интерфейса заключай в угловые кавычки (`«»`).

### 4.6.
Для путей в интерфейсе используй `→`.
Например: «Current User» → «Personal».

### 4.7.
Не используй жаргон.
Вместо этого используй официальные термины.

#### 4.7.1.
В частности, фразы в кавычках используй только в том случае, когда они являются точными цитатами.
Не используй фразы в кавычках для применения жаргонных фраз.
Например, следующий фрагмент текста недопустим, потому что там используется жаргонная фраза «пролетел»: 
```
Например, код, который пушит данные о покупке, подключён асинхронно и загружается с небольшой задержкой, а триггер уже «пролетел».
```

### 4.8.
Не используй «you need» и другие подобные обращённые к клиенту фразы, перекладывающие действия на него.
Помни: я пишу клиенту или потенциальному клиенту.
Делать в любом случае буду я, а не клиент.
Вместо «you need» используй 2 альтернативы:
#### 4.8.1.
Нейтральные фразы типа «it is necessary».
#### 4.8.2.
Глаголы в неопределённой форме.
Например, во фрагменте ниже использованы подобные глаголы «set up», «create»:
```
1.2) Set up the transfer of login events from WordPress to Power BI using Fabric / OneLake.
1.2.1) Set up a «Data Pipeline» from the WordPress database table that stores login events (see point 1.1) to Fabric / OneLake.
1.2.2) Set up a connection from Power BI to Fabric / OneLake to pass login events.
1.3) Create the data model in Power BI.
```
Обрати внимание, в этом фрагменте не говорится, кто именно будет выполнять описанные действия: ответственность не перекладывается на клиента, в отличие от «you need».

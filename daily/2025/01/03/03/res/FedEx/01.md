Нужно учитывать два важных момента:

1. **Официальные ограничения (throttling/usage limits).**  
   У FedEx (как и у других логистических компаний) существуют ограничения на количество запросов к API. Эти лимиты часто зависят от:  
   - Вашего тарифа (аккаунта) и соглашения с FedEx;  
   - Типа используемого API (Tracking, Rate, Ship и т. д.);  
   - Условий соглашения о коммерческом использовании (Commercial vs. Developer).  

   Как правило, информацию о лимитах запросов можно найти в **FedEx Developer Portal**. Если нужный лимит по умолчанию меньше, чем 100 000 запросов в желаемом периоде, FedEx может потребовать от вас:  
   - Увеличить лимиты через Enterprise-аккаунт или  
   - Обратиться к их технической поддержке за повышенной квотой.

2. **Практические рекомендации по массовым запросам.**  
   - **Объединять несколько трекинг-номеров в один запрос**, если API это поддерживает. У некоторых служб есть методы batch-запроса для трекингов: это позволяет «паковать» несколько номеров в одном запросе, снижая общее количество вызовов.  
   - **Разделить процесс на несколько «окон» или использовать систему очередей** (queue), чтобы не отправлять все 100 000 сразу, а распределить их (например, пачками по N трекингов).  
   - **Кэшировать результаты**: если часть номеров обновляется редко или обновление требует 1–2 запроса, возможно, имеет смысл не обращаться к API слишком часто.

Перед началом проекта стоит:
1. Связаться с поддержкой FedEx (или менеджером, если у вас коммерческий аккаунт) и уточнить актуальные ограничения по количеству запросов.  
2. Убедиться, что в конкретном API (Tracking API) есть функциональность, позволяющая передавать массив (batch) трекинг-номеров. Это обычно упрощает интеграцию при большом объёме.  
3. Настроить алгоритм распределения запросов или хранить/кэшировать полученные результаты, чтобы не «пересыщать» API неактуальными повторными вызовами.  

Таким образом, потенциально FedEx может разрешить 100 000 запросов (или даже больше), но ключ к этому — **иметь правильный аккаунт, согласованный объём запросов и грамотную схему вызовов (batch, очереди, кэширование)**. Без предварительного согласования, если просто отправить поток в 100 000 единичных запросов, велик риск получить отклонение части запросов или блокировку.

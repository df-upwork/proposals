1. Логическая ошибка (Стратегия): Несоответствие коммуникативной цели. `Aᨀ` представляет собой исчерпывающее техническое руководство, а не предложение о сотрудничестве (Proposal) на Upwork. Предоставление полного решения до начала контракта снижает мотивацию клиента к найму.
2. Логическая ошибка (Коммуникация): Поучающий тон. Формулировки «Ваше первое/второе возможное заблуждение» (п. 1, п. 2) могут быть восприняты как высокомерные и неуместны в начале общения с потенциальным клиентом.
3. Упущение (Стратегия): Игнорирование запроса на формат работы. Клиент просил «show me how to run it» (показать, как запустить), что подразумевает интерактивную сессию. `Aᨀ` предлагает только асинхронные инструкции, упуская предложение о звонке с демонстрацией экрана.
4. Критическое упущение (Безопасность): Отсутствие мер предосторожности перед тестовым запуском (п. 5). Запуск унаследованного конвейера без понимания его логики может повредить производственные данные. `Aᨀ` не включает обязательный шаг по анализу приемников данных (Sinks) и обеспечению запуска в безопасной (тестовой) среде или с использованием ограничений отладки.
5. Упущение (Логика процесса): Отсутствие этапа синтеза информации. План детально описывает шаги анализа (п. 1–4), но не включает явный этап обобщения результатов для выполнения запроса клиента «tell me what it is doing» (объяснить, что делает конвейер).
6. Упущение (Логика процесса): Незавершенность диагностики. Стратегическая задача клиента — понять причины прошлых сбоев. `Aᨀ` начинает диагностику (п. 1, 2), но затем переходит к запуску конвейера (п. 3–5) без этапа формулирования выводов о причинах исторических сбоев.
7. Фактическая ошибка/Заблуждение (п. 2.2.3): Избыточное доверие к «Test connection». Успешный тест подключения в ADF Studio не гарантирует работоспособность при запуске по триггеру. Тест выполняется в контексте пользователя, а производственный запуск — в контексте Managed Identity ADF. У них могут быть разные права доступа.
8. Упущение (Практика): Игнорирование прав доступа пользователя. Не учитывается, что у нового сотрудника может не хватать необходимых ролей Azure RBAC для выполнения предложенных действий, таких как «Test connection» (п. 2.2.3) или запуск «Debug» (п. 5.1).
9. Упущение (Техническое, п. 4): Неполный анализ источников параметров. Рассматриваются только триггеры. Упущена возможность вызова конвейера из другого конвейера через activity «Execute Pipeline», при котором параметры передаются родительским конвейером.
10. Упущение (Техническое, п. 5.1): Недостаточное рассмотрение режимов запуска. Предлагается только «Debug». Упущено упоминание «Trigger Now» и различие между ними: «Debug» тестирует текущую версию на холсте (возможно, неопубликованную), а «Trigger Now» — последнюю опубликованную (производственную) версию. Выбор режима зависит от цели тестирования.
11. Упущение (Техническое, п. 1.4.4): Не учтена специфика `SecureString`. Значения параметров типа `SecureString` (например, секреты из Key Vault) маскируются в журналах выполнения (Monitor) и не могут быть просмотрены напрямую.
12. Упущение (Техническое, п. 3.5): Неполный анализ использования параметров. Не упомянута необходимость анализа передачи и использования параметров внутри activity типа «Data Flow».
13. Упущение (Практика, п. 5.1): Игнорирование особенностей Data Flow. Если конвейер использует Data Flow, для его отладки требуется предварительная активация Data Flow Debug Session (запуск Spark-кластера), что занимает время.

# 12. Local Definitions
## 12.1.
~~~code
A[§P] ≔ B
~~~
Означает:
- Для понятия `B` я **временно**, **только в рамках** §`P`, использую обозначение `A`.
- Вне §`P` это правило не применяется: в частности, если до §`P` обозначение `A` имело другой смысл, то после §`P` обозначение `A` снова будет иметь этот смысл.
- По сути, `A[§P] ≔ B` объявляет **локальную переменную** `A` с **областью действия** §`P`.
- В отличие от `A[§P] ≔ B`, `A ≔ B` объявляет **глобальную переменную** `A`.

## 12.2.
~~~code
A[§P₁, §P₂, …, §Pₙ] ≔ B
~~~
Означает, что обозначение `A` имеет значение `B` в контексте ⠿{§`P₁`, §`P₂`, …, §`Pₙ`}.
По сути, это правило аналогично §12.1, но область действия локальной переменной `A` ограничивается не одним пунктом, а множеством пунктов.

## 12.3.
~~~code
A[§P₁-§Pₙ] ≔ B
~~~
Означает, что обозначение `A` имеет значение `B` в контексте ⠿{§P₁-§Pₙ}.
По сути, это правило аналогично §12.1 и §12.2.

## **2\. Анализ обоснованности: Проблемы миграции и нормализации унаследованных данных (Validating ≔†₁, ≔†₂, ≔†₃)**

Клиент (ꆜ) демонстрирует высокий уровень технической зрелости, точно идентифицируя существенные, нетривиальные проблемы в унаследованной (legacy) части проекта.

### **2.1. Обоснованность ≔†₁: Ручные процессы и миграция MS Access**

Обеспокоенность клиента миграцией "legacy Microsoft Access database" (O.md, PD2) и связанными с этим ручными процессами (≔†₁) полностью оправдана. Это не является тривиальной задачей "lift-and-shift".

* **Несоответствие типов данных:** Миграция требует тщательного сопоставления типов данных, которые не всегда имеют прямое соответствие. Например, тип данных Access Yes/No корректно преобразуется в тип bit в SQL Server 13, что может нарушить существующую логику, если она ожидает числовые (0/1) или текстовые (True/False) значения.  
* **Деградация производительности:** В PD2 указана задача "convert Access queries/macros to SQL views or stored procedures". Это критически важный и обоснованный шаг. Прямой перенос логики запросов из файловой архитектуры Access в клиент-серверную архитектуру Azure SQL часто приводит к катастрофическому падению производительности. Анализ показывает, что запросы, эффективно работавшие в Access (например, с использованием OUTER JOINs), становятся "очень медленными" (very slow) при выполнении на бэкенде Azure SQL.3 Единственным жизнеспособным решением является рефакторинг этой логики в хранимые процедуры или представления на стороне сервера, как и указано клиентом.3  
* **Рекомендуемая практика (SSMA):** Обоснованность подхода клиента подтверждается тем, что Microsoft предоставляет специализированный инструмент для решения именно этих проблем — **SQL Server Migration Assistant (SSMA) for Access**.14 PD2 корректно ссылается на "SSMA or equivalent tools". Роль SSMA заключается в автоматизации трех ключевых этапов, указанных в PD2:  
  1. **Оценка (Assessment):** SSMA генерирует отчеты, выявляющие проблемы конверсии и несоответствия типов.14  
  2. **Преобразование схемы (Schema Conversion):** SSMA автоматически преобразует объекты Access, включая запросы и макросы, в синтаксис Transact-SQL (представления или хранимые процедуры).14  
  3. **Миграция данных (Data Migration):** SSMA управляет непосредственно процессом переноса данных.14

### **2.2. Обоснованность ≔†₂ и ≔†₃: Рассогласование схем и идентификаторов (CGUID vs. Client ID)**

Проблема ≔†₃ ("Identifier mismatches: Legacy used CGUIDs; Eccovia issues Client IDs", O.md, PD2) является фундаментальной проблемой моделирования данных, и ее выявление клиентом свидетельствует о глубоком понимании предметной области.

* **Технический компромисс (GUID vs. INT):** Клиент столкнулся с классическим архитектурным компромиссом.  
  * **Обоснованность CGUID (Legacy):** GUID (глобальные уникальные идентификаторы) необходимы для систем, где данные создаются в *распределенных* или *офлайн* средах (что, вероятно, соответствует старой архитектуре Fountain House с Power Apps) и должны сливаться (merge) в центральную базу без конфликтов первичных ключей.16  
  * **Обоснованность Client ID (Eccovia):** GUID (16 байт) в четыре раза больше, чем INT (4 байта). Если GUID используется в качестве кластеризованного первичного ключа (что является распространенной практикой), его случайная природа приводит к *массивной фрагментации страниц индекса*.4 Это катастрофически снижает производительность операций INSERT и сканирования диапазона. Eccovia, как высокопроизводительная SaaS-платформа, с большой вероятностью использует INT (Identity) для обеспечения максимальной производительности.16  
* **Сложность решения:** Проблема ꆜ полностью обоснована. Решение требует создания сложных "золотых" таблиц сопоставления (mapping tables) для связи исторических CGUID с новыми Client ID из Eccovia, чтобы обеспечить целостность данных (≔†₂).  
* **Вложенная проблема: Нечеткое сопоставление (Fuzzy Matching):** PD2 также упоминает "fuzzy matching (e.g., name \+ DOB)" как часть нормализации. Это требование также обосновано и нетривиально. В интеграции данных здравоохранения 5, особенно при слиянии данных о пациентах, детерминированные совпадения (Name \= Name) недостаточны.  
  * Требуются вероятностные (probabilistic) подходы 18 для обработки опечаток, фонетических совпадений или разных форматов дат. Это подразумевает использование сложных алгоритмов, таких как Soundex (для фонетически схожих имен) или Jaro-Winkler (для опечаток).19 Это подтверждает, что задача "очистки" данных является сложным ETL-процессом.

### **2.3. Вывод по Разделу 2**

Опасения клиента (ꆜ) в отношении миграции унаследованных данных (Access, CGUID, Fuzzy Matching) полностью обоснованы. Клиент демонстрирует высокий уровень технической зрелости, точно определяя нетривиальные задачи, которые требуют специализированных инструментов (SSMA) и сложных ETL-процессов.

## **3\. Критический аудит архитектуры: Обоснование «нерешительности» Eccovia (Анализ ≔†₄)**

Этот раздел анализирует центральный конфликт проекта: предложение клиента использовать Elastic Query и "нерешительность" (hesitant) Eccovia (≔†₄). Анализ показывает, что позиция Eccovia является не только обоснованной, но и единственно верной с технической, юридической и нормативной точек зрения.

### **3.1. ≔†₄ как симптом: Фундаментальное непонимание Elastic Query**

"Нерешительность" Eccovia — это прямой результат того, что предложенное клиентом решение (Elastic Query) **неприменимо** для данного сценария использования.

* **Техническая необоснованность (Неверный сценарий):** Azure SQL Elastic Query (EQ) — это технология, разработанная для выполнения T-SQL запросов, *охватывающих несколько баз данных **Azure SQL Database***.8 Она предназначена для сценариев горизонтального (sharding) или вертикального секционирования *внутри* экосистемы Azure SQL.  
  * EQ **не является** универсальной заменой "связанных серверов" (Linked Servers) для подключения к произвольным внешним базам данных SQL, особенно к проприетарным, управляемым сторонним SaaS-поставщиком (Eccovia).6 База данных Eccovia ClientTrack — это не просто "еще одна Azure SQL DB" в подписке клиента; это изолированный, защищенный PaaS-ресурс вендора.  
* **Риски промышленной эксплуатации (Статус "Preview"):** Это наиболее критический технический блокиратор. Документация Microsoft указывает, что функция Elastic Query находится в состоянии **Preview** (предварительная версия).8  
  * Использование "Preview" функции для основной производственной интеграции, обрабатывающей **защищенную медицинскую информацию (PHI)**, является грубым нарушением лучших практик управления рисками. Microsoft не предоставляет соглашение об уровне обслуживания (SLA) и техническую поддержку для Preview-функций и оставляет за собой право изменять или удалять их в любой момент.9 "Нерешительность" Eccovia абсолютно оправдана; они не могут привязывать стабильность своей платформы к нестабильной, не поддерживаемой функции Azure.

### **3.2. Конфликт разрешений: Провал компромисса "Viewer-Level Access"**

Предложение Eccovia предоставить "viewer-level access as a compromise" (O.md, PD2) и вера клиента в то, что это может сработать, выявляют фатальное расхождение в требуемых и предлагаемых разрешениях.

1. **Шаг 1: Определение "Viewer-level".** В контексте SQL Server "viewer-level access" обычно подразумевает членство в роли db\_datareader, которая дает право SELECT для всех пользовательских таблиц.23  
2. **Шаг 2: Требования Elastic Query.** Для работы EQ необходимо выполнить T-SQL команды CREATE EXTERNAL DATA SOURCE 8 и CREATE EXTERNAL TABLE.25  
3. **Шаг 3: Анализ требуемых разрешений.** Документация Microsoft 21 и 25 однозначно указывает, что для CREATE EXTERNAL DATA SOURCE требуется разрешение ALTER ANY EXTERNAL DATA SOURCE.  
4. **Шаг 4: Оценка привилегий.** Разрешение ALTER ANY EXTERNAL DATA SOURCE *не является* "viewer-level". Источники характеризуют его как **"высокопривилегированное" (highly privileged)**.25 Причина в том, что оно дает возможность не только создавать и изменять *любой* внешний источник данных, но и, что критически важно, **предоставляет доступ ко *всем* учетным данным уровня базы данных (database scoped credentials)**, хранящимся в базе.25  
5. **Вывод:** Компромисс "viewer-level access" **технически несостоятелен** для реализации Elastic Query. Eccovia *физически не может* предоставить клиенту (ꆜ) привилегии уровня ALTER ANY EXTERNAL DATA SOURCE на своей производственной SaaS-базе данных. Это было бы равносильно передаче ключей администратора от базы данных и являлось бы вопиющим, недопустимым нарушением безопасности.

### **3.3. Аудит безопасности и соответствия HIPAA**

"Нерешительность" Eccovia (≔†₄) — это не просто технический, а юридический и нормативный императив, связанный с HIPAA.

* **Юридическая ответственность (HIPAA):** Eccovia, как поставщик EHR, выступает "Деловым партнером" (Business Associate) по смыслу HIPAA. Они несут прямую юридическую ответственность за обеспечение конфиденциальности, целостности и доступности PHI.26  
* **Обход контроля доступа:** Безопасность платформы Eccovia ClientTrack основана на **ролевой модели доступа (Role-Based Security)** на *уровне приложения*.27 Эта модель обеспечивает гранулированный контроль над тем, кто и какие записи пациентов может видеть, основываясь на рабочих группах, программах и т.д..29  
  * Прямой доступ к базе данных (через EQ) **полностью обходит (bypasses) всю эту критически важную логику безопасности на уровне приложения**. Пользователь с доступом EQ потенциально может прочитать *все* данные, включая те, которые ему не положено видеть, что является прямым нарушением принципа "минимальных привилегий" (least privilege) HIPAA.31  
* **Слабая аутентификация:** Еще один критический недостаток. Документация по EQ (в Preview) указывает, что она **не поддерживает аутентификацию Azure Active Directory (AAD)** / Managed Identity. Поддерживается **только SQL Authentication**.7  
  * Это означает, что высокопривилегированный логин и пароль к производственной базе данных Eccovia должны быть сохранены в виде DATABASE SCOPED CREDENTIAL 24 в базе данных клиента (ꆜ). Это значительно менее безопасный метод по сравнению с современными стандартами аутентификации (AAD) и создает дополнительный вектор атаки.

## **4\. Рекомендуемая архитектура: Жизнеспособные альтернативы для достижения ≔†₅ (Автономия данных)**

Вместо реализации рискованной и неработоспособной архитектуры Elastic Query, существуют три поддерживаемых и безопасных паттерна для достижения цели клиента (ꆜ) по созданию "analytics-ready data environment" (O.md, PD).

### **4.1. Паттерн 1 (Реал-тайм/Транзакционный): Интеграция через REST API (CTAPI)**

* **Решение:** Клиент (ꆜ) уже планирует использовать Azure Data Factory (ADF) (O.md, PD2), а Eccovia предоставляет зрелый REST API (CTAPI).10  
* **Реализация:** ADF имеет встроенный коннектор REST.32 Пайплайны ADF могут быть настроены для вызова эндпоинтов CTAPI (GET, POST, PUT).36  
* **Преимущества:**  
  1. **Безопасность:** Этот подход полностью соответствует модели безопасности Eccovia. Он использует аутентификацию на основе API-ключей. В частности, User Key ограничивает доступ к данным API точно так же, как если бы это был пользователь в интерфейсе, соблюдая все настроенные правила RBAC.10  
  2. **Поддержка Eccovia:** Это полностью документированный и поддерживаемый вендором путь интеграции.10  
  3. **Гибкость:** Подходит для запуска по триггеру или для небольших пакетных обновлений в реальном времени.  
* **Ограничение:** Может быть неэффективен для извлечения *массовых* исторических данных, о чем предупреждает сама Eccovia.37

### **4.2. Паттерн 2 (Пакетный/Аналитический): Асинхронный экспорт в Azure Storage**

* **Решение:** Этот паттерн напрямую решает проблему ≔†₁ ("manual weekly exports"). Eccovia предоставляет новую функцию "User-Specific Storage Container".11  
* **Процесс:**  
  1. Клиент (ꆜ) в интерфейсе ClientTrack создает "User-Specific Storage Container".11  
  2. Eccovia *асинхронно* экспортирует данные (например, по расписанию) в виде файлов (CSV, XML) в этот защищенный контейнер Azure Storage.11  
  3. Eccovia может вызвать Web Hook 11, который, в свою очередь, запускает пайплайн ADF.  
  4. ADF использует свой стандартный коннектор Azure Blob Storage 33 для загрузки этих файлов в целевую базу данных Azure SQL.  
* **Преимущества:**  
  1. **Идеально для аналитики:** Это *идеальное* решение для сценария клиента. Это эффективный, асинхронный, пакетный (bulk) экспорт, разработанный специально для замены ручных выгрузок.  
  2. **Безопасность:** Использует безопасные токены SAS (Shared Access Signature) для доступа к контейнеру, которые генерируются и контролируются клиентом.11  
  3. **Простота:** Архитектура (Storage \-\> ADF \-\> Azure SQL) является одним из самых распространенных, надежных и высокопроизводительных паттернов в Azure.38

### **4.3. Паттерн 3 (Стратегический/Долгосрочный): Интеграция на основе стандарта FHIR**

* **Стратегический инсайт:** Клиент (ꆜ) — это национальная некоммерческая организация в сфере здравоохранения 40 со своим исследовательским подразделением (RAKE) 1, которое отчитывается перед департаментами здравоохранения 2 и публикует исследования.41 Они решают проблему *медицинской* совместимости (interoperability), используя *общетехнический* инструмент (SQL). Это неверный подход.  
* **Решение:**  
  1. **Поддержка Eccovia:** Eccovia в своих материалах заявляет о поддержке **"HL7 FHIR-compliant data exchange"**.12 FHIR (Fast Healthcare Interoperability Resources) — это *глобальный стандарт* для обмена медицинскими данными.44  
  2. **Платформа Microsoft:** Microsoft предоставляет **Azure Health Data Services** 34, PaaS-решение (эволюция Azure API for FHIR 46), созданное для приема, хранения и анализа данных в формате FHIR.  
* **Предлагаемая архитектура:**  
  1. Вместо запроса SQL-доступа, ꆜ должен запросить у Eccovia эндпоинт FHIR.  
  2. Данные из Eccovia (через FHIR) и унаследованных систем (после преобразования с помощью инструментов, таких как FHIR Converter 47) загружаются в Azure Health Data Services.  
  3. Этот сервис становится *единым, стандартизированным источником правды* (single source of truth) для PHI.  
* **Преимущества:**  
  1. **Истинная автономия данных (≔†₅):** Это высшая форма автономии — не просто "сырые" таблицы, а *стандартизированные, семантически совместимые* данные.  
  2. **Соответствие целям ꆜ:** Эта архитектура *значительно* упростит аналитику, обмен данными и отчетность для исследовательской группы RAKE.  
  3. **Готовность к будущему:** Эта платформа нативно интегрируется с Azure Synapse, Power BI 48 и инструментами AI/ML.34

Анализ проекта (P⁎) приводит к следующим выводам:

2. **Обоснованная проблема (Интеграция):** Проблема клиента с интеграцией EHR (ручные процессы ≔†₁) **обоснована**.

### **5.2. План действий для исполнителя**

Исполнителю проекта (T⁎) не следует предлагать реализацию Elastic Query. Это приведет к техническому провалу, конфликту с Eccovia и создаст серьезные риски безопасности.

Вместо этого, исполнителю рекомендуется использовать данный аудит для *переориентации* клиента (ꆜ). Необходимо позиционировать себя не как простого исполнителя неработоспособного SOW, а как **стратегического партнера**, который провел due diligence, выявил критический архитектурный риск и предлагает безопасные, эффективные и соответствующие отраслевым стандартам решения.

Рекомендуемый SOW (Scope of Work) должен быть разделен на три фазы:

1. **Фаза 1: Миграция Legacy (Как в PD2).** Выполнить миграцию MS Access в Azure SQL с использованием SSMA.14 Основной фокус — на корректном преобразовании типов 13, рефакторинге логики запросов в хранимые процедуры 3 и создании таблиц сопоставления ID (CGUID \<-\> Client ID).16  
2. **Фаза 2: Интеграция (Пакетная).** Внедрить **Паттерн 2 (Async Export)**. Настроить Eccovia User Storage Container 11 и пайплайны ADF 33 для автоматизации еженедельной/ежедневной загрузки данных. Это немедленно решит проблему ≔†₁ и обеспечит "analytics-ready environment".  
3. **Фаза 3: Интеграция (Стратегическая).** Инициировать Discovery-сессию с ꆜ (включая RAKE) и Eccovia по **Паттерну 3 (FHIR)**.12 Это долгосрочное, стратегическое решение, которое наилучшим образом соответствует миссии Fountain House 1 и обеспечивает высшую степень "автономии данных" (≔†₅).

Анализ описания проекта (O.md, PD, PD2) и, в частности, выявленного конфликта между предлагаемой архитектурой и реакцией поставщика (O.md, ≔†₄: "Eccovia... hesitant"), позволяет идентифицировать следующий набор из четырех (4) основных технических заблуждений (D𐊑⠿), лежащих в основе стратегии клиента (ꆜ):

* D𐊑₂: «"Нерешительность" (hesitant) Eccovia (≔†₄) в предоставлении прямого доступа к SQL-базе является субъективным бизнес-препятствием (например, vendor lock-in), а не объективной технической и нормативной необходимостью».  
* D𐊑₃: «Компромисс в виде "доступа на уровне просмотра" (viewer-level access) технически достаточен для реализации Elastic Query».  
* D𐊑₄: «Прямой доступ к базе данных (EQ) является более предпочтительной или единственной жизнеспособной стратегией для достижения "автономии данных" (≔†₅) по сравнению с альтернативами (например, API или FHIR)».

## **2\. Детальный Анализ Заблуждений (D𐊑ᵢ)**

### **2.1. Анализ: D𐊑₁ (Заблуждение о пригодности и готовности Elastic Query)**

#### **2.1.1. Доводы за Pⰳ(D𐊑₁) (Этиология заблуждения ꆜ)**

Заблуждение клиента (ꆜ) о том, что Elastic Query (EQ) является подходящим решением, не является полностью необоснованным; оно, по-видимому, основано на поверхностном прочтении и критической неверной интерпретации высокоуровневой документации Microsoft.

Основная причина заблуждения кроется в маркетинговых описаниях EQ. Источники, такие как \[3\] и \[3\], описывают EQ как функцию, которая позволяет «выполнять... запросы T-SQL, охватывающие несколько баз данных» и, что наиболее важно, «подключать инструменты Microsoft и **сторонних производителей** (Excel, Power BI, Tableau и т. д.)».

Вероятная (но ошибочная) цепочка рассуждений клиента выглядит следующим образом:

1. Документация \[3\] упоминает «инструменты сторонних производителей».  
2. Клиент (ꆜ) экстраполирует это, предполагая, что если *инструменты* (клиенты запросов, такие как Tableau) могут быть сторонними, то и *источники данных* (целевые базы данных) также могут быть сторонними (т.е. принадлежать SaaS-поставщику, такому как Eccovia).  
3. Эта экстраполяция неверна. Документация описывает подключение BI-инструментов (Tableau) к *центральной базе данных Azure SQL, в которой настроен EQ* (т.е. к оркестратору запросов), а не подключение самого EQ к *произвольной сторонней базе данных* (Eccovia).  
4. Кроме того, клиент, вероятно, ищет облачную замену традиционным «связанным серверам» (Linked Servers) для выполнения межсерверных запросов. Описание EQ как «запросов, охватывающих несколько баз данных» \[3\] выглядит как современное PaaS-решение этой проблемы. Однако этот вывод преждевременен, поскольку другие источники (например, \[4\]) проводят четкое различие между EQ, Linked Servers (поддерживаемыми в Azure SQL Managed Instance) и другими подходами к интеграции.

Таким образом, этиология заблуждения D𐊑₁ — это добросовестная, но неверная интерпретация рекламных материалов, усугубленная смешением понятий «сторонний инструмент» и «сторонний источник данных».

#### **2.1.2. Доводы против Pⰳ(D𐊑₁) (Опровержение заблуждения)**

Заблуждение D𐊑₁ опровергается по трем независимым и фундаментальным направлениям. Каждое из них само по себе является достаточным основанием для признания архитектуры EQ нежизнеспособной для данного проекта.

**A. Критическое Несоответствие Сценария Использования**

Elastic Query не является универсальным инструментом интеграции. Анализ документации показывает, что он был разработан для двух очень специфических сценариев *внутреннего* масштабирования данных, контролируемых *самим клиентом*, а не для интеграции с *внешними* SaaS-платформами.

1. **Горизонтальное Секционирование (Шардинг):** Основное и наиболее документированное применение EQ — это запросы к данным, которые были горизонтально разделены (sharded) на множество баз данных Azure SQL. Источники \[3, 5, 6, 7\] многократно ссылаются на компоненты этой архитектуры, такие как "shard map" (карта шардов), "shard databases" (базы данных-шарды) и "elastic database client library" (клиентская библиотека эластичных баз данных). Это архитектура "scale-out", при которой одна логическая таблица (например, Customers) физически разделена между десятками или сотнями баз данных.  
2. **Вертикальное Секционирование:** Второе применение — это запросы к *разным* таблицам, находящимся в *разных* базах данных в *той же* среде клиента (например, таблица Customers в CustomerDB и таблица Orders в OrdersDB) \[3, 8\].  
3. **Управление:** Вспомогательная роль EQ проявляется в связке с *Elastic Jobs* (Эластичные Задания) для выполнения административных задач (например, сбор телеметрии, развертывание изменений схемы) в *коллекциях* баз данных, принадлежащих клиенту \[1, 9\].

Архитектура клиента (ꆜ) — это интеграция «точка-точка» (point-to-point): (Собственная Azure SQL DB) \<---\> (SaaS-вендор Eccovia). Предписанная архитектура EQ — это «один ко многим» (one-to-many) *внутри* собственной среды клиента: (Главная DB) \<---\> (Шард 1, Шард 2,..., Шард N).

Это фундаментальное несоответствие. EQ не является универсальной заменой «связанных серверов» \[2, 4\] и не предназначен для подключения к произвольным внешним базам данных SQL \[10, 11\], особенно к изолированной, защищенной PaaS-базе данных, управляемой сторонним SaaS-поставщиком.

**B. Фатальный Риск Эксплуатации: Статус "Perpetual Preview"**

Это наиболее критический блокиратор с точки зрения управления рисками. По состоянию на текущую дату (9 ноября 2025 г., согласно O.md §0), официальная документация Microsoft *по-прежнему* помечает функцию Elastic Query как **(in preview)** (предварительная версия) \[3, 8\].

Этот статус не является временным (например, новой бета-версией):

* Обсуждения в сообществе, датированные маем и июнем 2025 года \[12, 13\], подтверждают, что EQ «находится в статусе Preview... **уже несколько лет**» \[12\]. Пользователи выражают обеспокоенность тем, что функция «по-видимому, так и не вышла из состояния предварительной версии» \[13\].  
* Статус "Preview" имеет конкретные и серьезные последствия: отсутствие Соглашения об уровне обслуживания (SLA) и отсутствие полной производственной поддержки. Источники \[14\] и \[15\], описывая другие сервисы, четко проводят различие между функциями в "Public Preview" и общедоступными (GA) продуктами, которые «включают полную поддержку и SLA» \[14\].  
* Тот факт, что EQ остается в "Preview" \[12\], в то время как множество других сервисов и функций Azure *достигли* статуса General Availability (GA) в 2024 и 2025 годах \[16, 17, 18, 19, 20, 21\], является явным сигналом от Microsoft об отсутствии активных инвестиций в эту технологию и ее неясном будущем.

Клиент (ꆜ) работает в секторе здравоохранения (O.md, PD), обрабатывая Защищенную медицинскую информацию (Protected Health Information, PHI). Это среда с максимально высокими требованиями к стабильности, безопасности и поддержке. Предложение построить *основную* производственную аналитическую платформу на технологии, не имеющей SLA \[14\] и находящейся в состоянии "бессрочной предварительной версии" \[12\], является грубым нарушением базовых принципов управления техническими рисками. "Нерешительность" Eccovia (≔†₄) в этом свете является не препятствием, а выполнением того due diligence, который клиент (ꆜ) не смог выполнить.

**C. Критический Блокатор: Отсутствие Безопасной Аутентификации**

Это наиболее фатальный *технический* недостаток, который делает предложенную архитектуру неработоспособной "на старте" (dead on arrival).

Официальная документация Microsoft по EQ содержит недвусмысленное и фатальное ограничение: «Elastic queries **only supported** when connecting with **SQL Server Authentication**» (Эластичные запросы поддерживаются *только* при подключении с использованием *аутентификации SQL Server*) \[3\].

Другие источники подтверждают это:

* EQ «Does not support Azure Managed Identity» (Не поддерживает Управляемое удостоверение Azure) \[12\].  
* «Azure Active Directory (AAD) principals are not supported... Managed Identity... is also not supported» (Принципалы AAD не поддерживаются... Управляемое удостоверение также не поддерживается) \[22\].

Это ограничение напрямую противоречит *всем* современным лучшим практикам безопасности в Azure, которые настоятельно предписывают использовать аутентификацию Microsoft Entra ID (ранее AAD) и Управляемые удостоверения (\[21, 23, 24, 25\]) именно для того, чтобы *избежать* использования и хранения паролей.

Ирония заключается в том, что даже родственная технология *Elastic Jobs* (Эластичные Задания) *была обновлена* и *поддерживает* аутентификацию через Управляемое удостоверение (UMI) \[26\]. Тот факт, что *Elastic Query* не получил этого обновления, еще раз подтверждает его статус заброшенной технологии (как обсуждалось в пункте B).

Последствия этого ограничения для проекта P⁎ катастрофичны:

1. Клиент (ꆜ) просит Eccovia (стороннего SaaS-вендора) о прямом SQL-доступе.  
2. Из-за фундаментального ограничения EQ \[3, 22\], ꆜ вынужден запрашивать у Eccovia *не* современный гостевой доступ AAD, а **SQL-логин и пароль** (синтаксис в \[7\] подтверждает это: WITH IDENTITY \= '\<username\>', SECRET \= '\<password\>').  
3. Ни один компетентный SaaS-вендор, особенно в сфере здравоохранения, соответствующей HIPAA, в 2025 году *никогда* не предоставит внешнему субъекту SQL-логин и пароль к своей производственной многопользовательской базе данных. Это было бы равносильно передаче ключей от всего хранилища данных.  
4. Следовательно, предложение ꆜ (Elastic Query) технически нежизнеспособно, поскольку оно требует устаревшего и небезопасного протокола аутентификации, который не будет (и не должен) поддерживаться вендором.

#### **2.1.4. Вердикт по D𐊑₁**

Заблуждение D𐊑₁ полностью подтверждено (Pⰳ(D𐊑₁) \= 100). Технология Azure SQL Elastic Query **категорически непригодна** для проекта P⁎. Она не соответствует сценарию использования (предназначена для *внутреннего* шардинга \[3, 7\], а не для *внешней* SaaS-интеграции), представляет собой неприемлемый производственный риск (многолетний статус "Preview" без SLA \[12, 13\]) и, что является решающим фактором, технически неработоспособна из\-за отсутствия поддержки современной аутентификации (AAD/Managed Identity) \[12, 22\], требуя небезопасного и неприемлемого обмена SQL-паролями \[3\].

### **2.2. Анализ: D𐊑₂ (Заблуждение о «нерешительности» Eccovia)**

#### **2.2.1. Доводы за Pⰳ(D𐊑₂) (Этиология заблуждения ꆜ)**

Восприятие клиентом (ꆜ) "нерешительности" Eccovia (≔†₄) как субъективного бизнес-препятствия понятно в контексте стандартной динамики «клиент-вендор». Клиент придерживается (обоснованного) стратегического принципа "автономии данных" (≔†₅, O.md, PD2). С этой точки зрения, любые действия поставщика, ограничивающие доступ к данным, которые клиент считает *своими*, инстинктивно воспринимаются как враждебный акт, направленный на "vendor lock-in" (удержание клиента), принуждение к покупке дорогостоящих API-вызовов или просто нежелание сотрудничать. Это стандартная точка зрения клиента, не обладающего полной информацией о технических и нормативных ограничениях поставщика.

#### **2.2.2. Доводы против Pⰳ(D𐊑₂) (Опровержение заблуждения)**

"Нерешительность" Eccovia (≔†₄) не является субъективной бизнес-тактикой. Напротив, анализ показывает, что это *объективная, необходимая и единственно ответственная* реакция, продиктованная двумя непреодолимыми императивами: (A) нормативно-правовыми обязательствами (HIPAA) и (B) фундаментальными принципами технической безопасности (недопустимость обхода RBAC).

**A. Нормативно-Правовой Императив (HIPAA)**

Eccovia, как поставщик EHR-системы (O.md, PD), обрабатывающей данные о психическом здоровье, юридически классифицируется как «Деловой партнер» (Business Associate) в соответствии с Законом о переносимости и подотчетности медицинского страхования (HIPAA). В этом качестве Eccovia несет прямую юридическую ответственность за обеспечение конфиденциальности, целостности и доступности PHI \[27\].

Закон HIPAA — это не просто рекомендация; он устанавливает строгие стандарты безопасности. Ключевым среди них является «Правило безопасности» (Security Rule), которое требует:

* Обеспечение конфиденциальности, то есть «данные... не предоставляются... неавторизованным лицам или процессам» \[28\].  
* Реализация «политик и процедур для санкционирования доступа к ePHI **только тогда, когда такой доступ соответствует роли пользователя**» \[28\].  
* Соблюдение стандарта «минимально необходимого» (minimum necessary) \[28\].

На практике этот гранулированный контроль доступа реализуется через модель Ролевого Контроля Доступа (Role-Based Access Control, RBAC) \[29, 30\]. В секторе здравоохранения \[30\] RBAC — это не просто функция, а фундаментальный компонент соответствия требованиям.

Документация по продукту Eccovia ClientTrack (представленная в источниках \[31\] \- \[32\]) подтверждает, что их система основана на сложной, гранулированной модели безопасности, управляемой *на уровне приложения*. Разрешения определяются на основе "Workgroup Access" (Доступ к рабочей группе), "Program Management" (Управление программой) и "Organization Access" (Доступ к организации) \[33\]. В \[34\] подробно описаны "Tool Access Permissions" (Разрешения на доступ к инструментам), что подтверждает наличие сложной логики авторизации, встроенной в само приложение.

Запрос клиента (ꆜ) на прямой доступ к базе данных (через Elastic Query) представляет собой экзистенциальную угрозу этой модели:

1. Модель безопасности Eccovia \[34\] гарантирует, что пользователь A (например, социальный работник) может видеть только записи пациентов, относящиеся к его "Рабочей группе".  
2. Запрос ꆜ (EQ) — это SELECT на уровне *всей базы данных*, выполняемый *под* уровнем приложения.  
3. Этот SELECT **полностью обходит (bypasses)** всю критически важную логику RBAC на уровне приложения \[35\].  
4. В результате учетная запись, используемая ꆜ для EQ, получила бы доступ ко *всем* записям пациентов во *всех* рабочих группах, включая те, которые ей не положено видеть по роли.  
5. Это является прямым и вопиющим нарушением принципа "минимально необходимого" HIPAA \[28\].

Таким образом, "нерешительность" Eccovia — это их юридический отдел, *запрещающий* ИТ-отделу выполнять запрос, который равносилен *соучастию в нарушении HIPAA*.

**B. Императив Технической Безопасности (Риски "Read-Only")**

Клиент (ꆜ) может возразить, что им нужен только "read-only" (только для чтения) доступ, который (по их мнению) безопасен и не несет рисков. Этот аргумент демонстрирует фундаментальное непонимание модели угроз при работе с PHI.

* Источник \[35\] идеально отражает этот спор: «...старший коллега выступает за \[прямой доступ к БД с read-only\]... Я выразил обеспокоенность, такую как:...Риски безопасности: **даже read-only доступ может раскрыть конфиденциальные данные**, если учетные данные будут... использованы не по назначению... **Обход бизнес-логики**...».  
* В секторе здравоохранения \[30\] или при работе с любыми конфиденциальными данными, *несанкционированное чтение (раскрытие)* часто является *более серьезным* нарушением, чем несанкционированная *запись (изменение)*. В то время как в банковской сфере \[30\] основной риск — это несанкционированный UPDATE (изменение баланса счета), в здравоохранении \[30\] основной риск — это несанкционированный SELECT (кража медицинских записей знаменитости, несанкционированное исследование, раскрытие диагнозов).  
* Аргумент клиента "это всего лишь read-only" доказывает, что он неверно оценивает риски. Eccovia, как ответственный хранитель PHI, *не может* позволить обойти свою логику безопасности и аудита на уровне приложения \[35\].  
* Дополнительные риски прямого доступа к БД, упомянутые в \[36\], включают невозможность SSO и использование слабых паролей, что усугубляется требованием EQ использовать *только* SQL Auth \[3\], как обсуждалось в D𐊑₁.

"Нерешительность" Eccovia — это признак зрелой организации безопасности, которая понимает, что в контексте HIPAA "чтение" *является* основной угрозой. Их позиция — это защита *своего* продукта, *своей* юридической ответственности и, в конечном счете, *самого клиента* (ꆜ) от катастрофического нарушения данных.

#### **2.2.3. Оценка Pⰳ(D𐊑₂)**

3. Следовательно, "viewer" доступ достаточен для выполнения "query" (запроса).

Это предположение, вероятно, было сделано менеджером проекта со стороны ꆜ и не прошло техническую проверку.

Шаг 1: Определение "Viewer-level access" (Компромисс Eccovia)  
В контексте SQL Server, "viewer-level access" (доступ на уровне просмотра) или "Data viewer" (просмотрщик данных) однозначно соответствует встроенной роли базы данных db\_datareader \[37, 38, 39, 40, 41, 42\]. Шаг 2: Определение требований D𐊑₁ (Решение ꆜ)  
Как установлено в D𐊑₁, для реализации Elastic Query клиент (ꆜ) должен выполнить T-SQL команды CREATE EXTERNAL DATA SOURCE (Создать внешний источник данных) \[3, 6, 8\] и CREATE EXTERNAL TABLE (Создать внешнюю таблицу) \[3, 43, 44\].

Официальная документация Microsoft однозначно указывает, какие разрешения требуются для этих команд DDL. В частности, для выполнения CREATE EXTERNAL DATA SOURCE требуется разрешение ALTER ANY EXTERNAL DATA SOURCE \[44, 45, 46, 47, 48\]. Источник \[8\] прямо указывает это разрешение в разделе "Prerequisites" (Обязательные условия) для настройки EQ.

Возникает фатальное несоответствие:

* **Требуется** для EQ: ALTER ANY EXTERNAL DATA SOURCE \[8\].  
* **Предложено** Eccovia: db\_datareader \[38\].

Эти разрешения находятся на противоположных концах спектра привилегий. db\_datareader — это *минимальное* разрешение на чтение. ALTER ANY EXTERNAL DATA SOURCE — это *высокопривилегированное* разрешение DDL \[10, 46, 49\]. Попытка клиента (ꆜ) выполнить CREATE EXTERNAL DATA SOURCE с правами db\_datareader немедленно завершится ошибкой PERMISSION DENIED.

"Нерешительность" Eccovia становится еще более понятной, если проанализировать, что на самом деле делает разрешение, которое действительно нужно клиенту (ALTER ANY EXTERNAL DATA SOURCE).  
Источник \[46\] (официальная документация Microsoft) содержит "дымящийся пистолет" (smoking gun) — прямое предупреждение о безопасности:

Это разрешение должно рассматриваться как высокопривилегированное (highly privileged) и должно предоставляться только доверенным принципалам в системе».  
(Другие источники, такие как \[10, 42, 50, 51, 52, 53, 54, 55\], подтверждают, что DATABASE SCOPED CREDENTIALS являются стандартным механизмом для хранения секретов (ключей API, паролей) для доступа к внешним данным, что многократно усиливает критичность риска, описанного в \[46\]).

Из этого следует очевидная логическая цепочка:

1. Клиент ꆜ запрашивает возможность использовать EQ.  
2. Это *неявно* является запросом на разрешение ALTER ANY EXTERNAL DATA SOURCE \[8\].  
3. ИТ-отдел Eccovia видит, что это "высокопривилегированное" разрешение \[46\], которое дает доступ ко *всем* секретам, хранящимся в их производственной базе данных.  
4. Предоставление этого разрешения внешнему субъекту равносильно передаче ключей администратора и является недопустимым нарушением безопасности.  
5. Eccovia *не может* предоставить это разрешение. Вместо этого они предлагают "вежливый отказ", который клиент не понял: «Мы не можем этого сделать. Максимум, о чем *теоретически* можно говорить — это "viewer-level access" (db\_datareader)».  
6. Клиент (ꆜ), не проверив технические требования \[8\], ошибочно принимает этот отказ за работающий компромисс.

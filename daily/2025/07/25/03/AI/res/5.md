

# **Анализ Внутреннего Языка (KL) Платформы KOIOS Master Data**

## **Введение**

Настоящий отчет представляет собой исчерпывающий технический анализ внутреннего формального языка, обозначаемого как KL, который используется платформой K (KOIOS Master Data). Первоначальный анализ показывает, что KL не является языком программирования в традиционном понимании, таким как процедурные или объектно-ориентированные языки (например, Java или Python). Вместо этого, KL представляет собой комплексную декларативную систему для моделирования, спецификации и обмена мастер-данными. «Язык» этой системы состоит из концептуальных моделей, семантических структур и синтаксических правил, определенных набором международных стандартов, в первую очередь сериями ISO 8000, ISO 22745 и ISO 29002\.

Цель данного отчета — предоставить исчерпывающую деконструкцию этой системы. Анализ разделен на три основных уровня, составляющих KL:

1. **Концептуальный уровень**: Философские и архитектурные принципы, вытекающие из стандарта ISO 8000, которые управляют качеством данных и лежат в основе всей платформы.  
2. **Лексико-синтаксический уровень**: Словарь, семантика и грамматика, определяемые стандартами ISO 22745 (Открытые технические словари) и ISO 29002 (Обмен данными характеристик), которые физически проявляются в виде специфической схемы XML.  
3. **Прагматический (интеракционный) уровень**: Практическое применение и манипулирование этой системой через пользовательский интерфейс платформы KOIOS и, что более важно для системной интеграции, через ее программный интерфейс REST API.

Методология анализа основана на синтезе общедоступной технической документации, включая определения услуг, спецификации API и презентации основателя компании, Питера Илза (Peter Eales), который является признанным экспертом в данной области.1

## **Раздел 1: Архитектурный план: ISO 8000 как основа KL**

Философия проектирования KL является прямой реализацией принципов, заложенных в международном стандарте качества данных ISO 8000\. Многочисленные документы, от определений услуг до технических описаний, многократно подчеркивают, что платформа KOIOS построена на архитектуре ISO 8000\.2 Презентации основателя компании Питера Илза служат не просто маркетинговыми материалами, а техническим руководством по философии стандарта.5 В них подчеркивается, что качество данных должно обеспечиваться на «этапе создания» и по принципу «снизу вверх», начиная с «наименьшего значимого элемента — значения свойства». Этот подход напрямую определяет структуру

KL, которая спроектирована для обеспечения такой гранулярности. Таким образом, понимание KL невозможно без предварительного понимания мировоззрения ISO 8000\.

### **1.1. Принцип 1: Машиночитаемые данные в сравнении с цифровыми данными**

Ключевым для понимания KL является различие, которое стандарт ISO 8000 проводит между просто цифровыми данными (например, таблица в PDF-файле) и подлинно машиночитаемыми данными.5

KL предназначен исключительно для работы с последними. Его цель — структурировать данные в формате, который может быть «легко обработан компьютером без вмешательства человека, при этом гарантируя, что семантическое значение не будет потеряно».5 Этот принцип диктует выбор формального синтаксиса (например, XML или JSON) и отказ от неструктурированных или полуструктурированных форматов. В то время как PDF-документ является цифровым, он не является машиночитаемым, поскольку компьютеру сложно извлечь табличную информацию, даже если она легко читается человеком.5

### **1.2. Принцип 2: Подход «снизу вверх» к качеству данных**

В отчете подробно рассматривается подход ISO 8000 к построению качества, начиная с «наименьшего значимого элемента — значения свойства».5 Это объясняет, почему

KL является настолько гранулярным. Платформа KOIOS уделяет особое внимание управлению «списками значений, единицами измерения и квалификаторами измерений» 3, что является прямым следствием этого принципа. Качество записи мастер-данных является функцией качества составляющих ее значений свойств.

### **1.3. Ключевые архитектурные компоненты KL (согласно ISO 8000-1)**

Архитектурная схема, представленная Питером Илзом, раскрывает foundational-элементы системы KL.5 Каждый компонент играет определенную роль в общей структуре:

* **Словарь данных (Data Dictionary)**: Семантическое ядро системы. Он определяет концепты, терминологию и является источником истины для значения данных. В KL этот компонент реализован как Открытый технический словарь (Open Technical Dictionary, OTD) на основе стандарта ISO 22745\. Платформа KOIOS поставляется с предварительно заполненным словарем, содержащим более 100 000 терминов из авторитетных источников.3  
* **Спецификация данных (Data Specification)**: Структурный шаблон. Он определяет необходимые свойства, правила и ограничения для описания объекта, используя концепты из Словаря данных. В KL это то, что пользователи настраивают как «требование к данным» (data requirement).2 KOIOS предоставляет библиотеку из более чем 40 000 готовых спецификаций.3  
* **Схема идентификации (Identification Scheme)**: Система глобально уникальных имен. Она предоставляет разрешаемые идентификаторы для каждого элемента (концептов, свойств, объектов). В KL это реализуется с помощью Международных регистрационных идентификаторов данных (International Registration Data Identifiers, IRDI), как указано в ISO/TS 29002-5.7  
* **Формальный синтаксис (Formal Syntax)**: Физический формат кодирования. Он определяет, как данные записываются в файл для обмена. В KL в качестве синтаксиса указаны XML 5 и JSON 3, а детальная структура регулируется стандартом ISO 29002-10.

## **Раздел 2: Формальный синтаксис и лексикон KL: Анализ ISO 22745 и ISO 29002**

Этот раздел переходит от концептуального «почему» к конкретному «что», предоставляя детальный анализ структуры и грамматики KL. Анализ демонстрирует, что KL не является проприетарным изобретением, а представляет собой строгую и последовательную реализацию общедоступных международных стандартов. Компания KOIOS Master Data многократно ссылается на ISO 22745 и ISO 29002 как на основу своей платформы.3 Обнаруженное руководство eCl@ss по встраиванию данных в XML-формат ISO 29002-10 9 по сути является «исходным кодом» синтаксиса

KL. Анализ этой структуры позволяет реконструировать точные правила языка.

### **2.1. Лексикон и семантика: Открытые технические словари (OTD) по ISO 22745**

Лексикон KL организован в соответствии со стандартом ISO 22745, который определяет структуру Открытых технических словарей (OTD). OTD функционирует как репозиторий концептов, где каждый концепт объединяет синонимичные термины, определения и изображения.10 Платформа KOIOS реализует это, позволяя подписчикам создавать собственный «корпоративный словарь», сопоставляя свою внутреннюю терминологию со стандартными концептами в OTD KOIOS.3 Этот процесс сопоставления является ключевым «актом программирования» в рамках

KL. Кроме того, стандарт ISO 22745-14 (Интерфейс запросов к словарю) 10 можно рассматривать как стандартизированного предшественника проприетарного REST API компании KOIOS.

### **2.2. Физический синтаксис: формат обмена XML по ISO 29002-10**

Этот раздел представляет собой техническое ядро отчета и содержит глубокий анализ структуры XML, которая составляет документ KL. Анализ основан преимущественно на руководстве по реализации eCl@ss.9

* **Общая структура документа**: Корневым элементом является \<cat:catalogue\>, который содержит пространства имен для различных частей стандарта и ссылку на схему XSD. Внутри него располагаются один или несколько элементов \<cat:item\>, представляющих объекты мастер-данных.  
* **Идентификация объекта**: Каждый \<cat:item\> идентифицируется через обязательный атрибут class\_ref. Значением этого атрибута является Международный регистрационный идентификатор данных (IRDI), например 0173-1\#01-AAA123\#001, который уникально ссылается на класс объекта в словаре.7 Это обеспечивает глобальную уникальность и разрешимость ссылок.  
* **Представление свойство-значение**: Данные внутри объекта представлены элементами \<cat:property\_value\>. Каждый такой элемент ссылается на конкретное свойство через атрибуты property\_ref и subitem\_path\_property\_ref, значениями которых также являются IRDI.  
* **Типизация данных**: ISO 29002-10 определяет строгую и подробную XML-структуру для различных типов данных. В таблице 1 представлено сопоставление логических типов данных с их конкретным синтаксисом в XML, что является практическим руководством для разработчиков, которым необходимо генерировать KL-совместимые данные.

## **Раздел 3: Интеракционный уровень KL: Платформа и API KOIOS**

Этот раздел демонстрирует, как KL используется на практике, связывая абстрактные стандарты с конкретными инструментами, предоставляемыми KOIOS. Определение услуги 2 описывает систему, настраиваемую пользователем, а страница технических преимуществ 3 и документация API 13 объясняют, как это достигается. Пользователь настраивает модели данных через платформу, которые затем можно создавать, обновлять и запрашивать программно через REST API. Конечные точки API напрямую соответствуют основным объектам системы

KL (концептам, каталогам). Проект клиента P 14 направлен на создание нового, управляемого ИИ инструмента для автоматизации части этого рабочего процесса, а именно «представления концептов», что напрямую соответствует вызову API

POST /concept.

### **3.1. Декларативное моделирование через платформу KOIOS**

Этот подраздел описывает пользовательский аспект KL. Анализируется, как пользователь через интерфейс KOIOS выполняет моделирование данных путем:

* Создания «требований к данным», которые являются функциональными эквивалентами существующих форм или документов.2  
* Определения «корпоративного словаря» путем добавления собственных терминов и их сопоставления с существующими концептами в OTD.3  
* Выбора из более чем 40 000 существующих «спецификаций данных» и их кастомизации.3

Этот процесс, по сути, является высокоуровневым, декларативным «программированием» в рамках системы KL.

### **3.2. Программное манипулирование через REST API**

Анализ документации API, доступной по адресу koios.docs.apiary.io 13, показывает, что это основной интерфейс для межсистемного взаимодействия со структурами

KL.

* **Аутентификация**: Для доступа к большинству конечных точек требуется access\_token в качестве параметра запроса и заголовок Client-ID.13  
* **Основные ресурсы**: Анализ фокусируется на двух основных ресурсах, которые отражают структуру стандартов ISO:  
  * **/concept**: Конечная точка для взаимодействия со Словарем данных (OTD). GET /concept извлекает концепты, а POST /concept создает новые. Это программная точка входа для проекта клиента P.  
  * **/catalogue**: Конечная точка для взаимодействия с записями мастер-данных (объектами). GET /catalogue/{id} извлекает объект, POST /catalogue создает новый, а PUT /catalogue/{id} обновляет существующий. Эти действия соответствуют созданию и управлению экземплярами спецификаций данных.  
* **«Разрешимая» природа KL**: Когда система получает XML-документ KL, она может использовать конечные точки API, такие как GET /concept/{id}, для получения полного, актуального семантического определения любого идентификатора (IRDI) в документе. Эта услуга по разрешению идентификаторов предоставляется бесплатно.3 Это превращает  
  KL из статичного формата данных в динамическую, веб\-ориентированную систему, схожую по принципам с Linked Data.

## **Раздел 4: Синтез: Целостное определение языка KOIOS (KL)**

Синтезируя результаты предыдущих разделов, можно дать комплексное, многоуровневое определение KL. Это не единый артефакт, а **трехуровневая формальная система для представления и обмена мастер-данными**:

1. **Концептуальный уровень (Философия «Почему»)**: Основан на ISO 8000, устанавливает непреложные принципы качества данных, машиночитаемости и семантической точности. Это идеологическая основа системы.  
2. **Логико-синтаксический уровень (Структура «Что»)**: Основан на ISO 22745 и ISO 29002, предоставляет конкретный лексикон (OTD) и грамматику (схему XML) для физического представления данных. Это формальная грамматика языка.  
3. **Прагматический уровень (Механизм «Как»)**: Реализован через платформу KOIOS и REST API, предоставляет человеческие и программные интерфейсы для создания, управления и разрешения структур данных, определенных на нижних уровнях. Это среда выполнения и взаимодействия.

Сила KL заключается в сочетании стандартизированного, статичного синтаксиса (XML) с динамическим механизмом разрешения в реальном времени (API), что делает его практической реализацией принципов Linked Data для управления промышленными мастер-данными.

### **A.1. Устранение неоднозначности термина «Koios»**

Следует отметить, что данный анализ относится исключительно к компании **KOIOS Master Data Limited** (Великобритания, основатель Питер Илз), специализирующейся на решениях на базе ISO 8000\.15 В ходе исследования были выявлены другие, не связанные с ней сущности с похожим названием, которые необходимо разграничить для ясности:

* **Koios API**: Децентрализованный RESTful-интерфейс для блокчейна Cardano.17  
* **Koios Cloud Platform**: Платформа для управления энергосистемами от компании Sparkmeter.18  
* **KOIOS**: Инструмент для идентификации и картирования геномных вариантов от Odysseus Data Services.19

### **A.2. Примечание о статусе компании**

В ходе исследования была обнаружена противоречивая информация о статусе компании KOIOS Master Data Limited. Один из агрегаторов данных указывает на статус «Deadpooled» (прекратила деятельность).15 Однако официальные данные Регистрационной палаты Великобритании (Companies House) 16, а также недавнее участие компании в работе комитетов по стандартизации BSI Group 21, свидетельствуют о том, что она является действующей, хотя и небольшой, узкоспециализированной организацией.

2. **Логическая (техническая) ошибка в пункте 8.3.**
Рекомендация опустить параметр `directConnection` (`DC`) для того, чтобы драйвер использовал обнаружение топологии через Private DNS, является технически некорректной для архитектуры с Private Service Connect (PSC).
Эндпоинт PSC предоставляет единый IP-адрес (или малый набор адресов), действующий как балансировщик нагрузки. Если драйвер выполнит обнаружение топологии, он попытается установить соединения с конкретными узлами кластера по их DNS-именам. Даже если Private DNS разрешит все эти имена в IP-адрес PSC, без гарантии корректной маршрутизации по SNI на стороне инфраструктуры Atlas PSC (что не всегда гарантируется для всех типов эндпоинтов), запрос драйвера к конкретному узлу может быть направлен балансировщиком на другой узел. Это приведет к ошибке протокола (mismatch me field) и разрыву соединения. Для стабильной работы через PSC рекомендуется использовать `directConnection=true`, чтобы драйвер работал с эндпоинтом как с единой точкой входа.

3. **Логическая ошибка в пункте 8.4.**
В тексте утверждается: «Enable «Private Google Access»... The workers will naturally utilize your Private DNS Zone...». Хотя само действие по включению Private Google Access (PGA) необходимо для работы Dataflow (для доступа к API Google без внешних IP), обоснование, связывающее его с работой Private DNS, может ввести в заблуждение. Воркеры используют Private DNS Zone не из-за включения PGA, а благодаря своему расположению внутри VPC сети и использованию внутреннего метадата-сервера. PGA влияет только на маршрутизацию трафика к публичным API Google (Cloud Storage, BigQuery), но не на разрешение доменных имен MongoDB.

~~~
Судя по описанию, основная причина в том, что в **Tenant 1** (где создаётся встреча в Teams) **нет нужного механизма** или **не настроен процесс** для автоматического создания (или синхронизации) учётных записей пользователей из **Tenant 2**. 

Другими словами, когда организатор из Tenant 1 пытается назначить приглашённого пользователя из Tenant 2 сопроводителем (co-organizer), в **Azure AD** первого арендатора не существует «гостевой» записи для этого пользователя. Из‐за этого Microsoft Teams не может присвоить ему права, необходимые для роли организатора, ведь Teams «под капотом» полагается на Azure AD (или на сопоставление в «локальную» для арендатора директорию).

При этом **междоменный чат** (внешний федеративный чат) может работать, потому что для текстовой коммуникации требуется лишь включённая федерация между доменами/арендаторами (внешние чаты не создают полноценной учётной записи в чужом Azure AD). Но для роли «co-organizer» нужно именно существование пользователя (в виде guest или другого типа учётной записи) в Tenant 1. 

Таким образом, кратко:
1. **Чат работает**, потому что включена федерация между доменами в Teams, и для этого не нужно создавать учётную запись во внешнем арендаторе.  
2. **Назначение сопроводителем (co-organizer) не работает**, поскольку в Azure AD Tenant 1 не появляется запись пользователя из Tenant 2, следовательно, Teams не может назначить ему необходимые права в своей среде.
~~~

I. Notes:
1) open.epic.com is primarily intended for FHIR integrations (REST API) and does not provide public access to HL7 v2.
2) Large Epic partners can negotiate with Epic Systems for an environment where HL7 v2 traffic is possible (although limited).
3) Hospitals that are Epic customers almost always have a staging installation.
There, it is possible to configure Epic Bridges to send HL7 v2 ADT^A03 (discharge) to Mirth Connect (TCP/MLLP).
---
II. The right way to solve your task (via Epic Bridges):
1) Create a new channel in Mirth Connect.
2) Configure the Mirth Connect Source Connector to receive HL7 messages:
2.1) In the «Source» section → «Connector Type», select «TCP Listener» or «MLLP Listener» (most commonly MLLP Listener for HL7 v2.x).
2.2) Specify the port on which Mirth Connect will receive incoming HL7 messages.
2.3) In «Receive Timeout» and «Buffer Size», set parameters according to project requirements or Epic recommendations.
2.4) In «Response», select «Auto-generate ACK» (if sending an acknowledge is required).
3) If necessary, add a Transformer in Mirth Connect to pre-process incoming HL7 messages:
3.1) In the «Source» → «Edit Transformer» block, it is possible to use the built-in HL7 v2.x parser.
3.2) Extract the required fields (e.g. `PID.3` for the patient identifier).
When using JavaScript, it is possible to refer to segments like `msg['PID']['PID.3']['PID.3.1']`.
3.3) If desired, transform the message format (e.g. remove unnecessary segments or write separate fields into temporary variables).
4) In the «Destinations» section, create a «Destination Connector» of the «Channel Writer» type for an initial view or logging.
5) Set up the connection on the Epic EHR side through Epic Bridges:
5.1) In Epic Bridges, create an interface to send HL7 messages (ADT, ORM, ORU, etc.) to the IP address and port where the «MLLP Listener» listens for messages.
5.2) Specify the event types (`ADT^A03` for discharge) and the necessary segments.
5.3) Save the configuration and restart the Bridges service.
6) Check the channel operation in the Mirth Connect Administrator «Dashboard»:
6.1) Go to «Dashboard» → «Channels».
6.2) Locate the created channel and click «Deploy Channel».
6.3) Verify that the channel status is «Deployed» with no errors.
7) Check the connection between Epic Bridges and Mirth Connect:
7.1) Send a test HL7 message from Epic Bridges.
7.2) In the Mirth Connect Administrator interface, open «Dashboard» → «Messages» for the respective channel and verify that the message was received and processed without errors.
7.3) If necessary, view the response (ACK) in the «Message Browser».
8) Verify that the database is ready to receive logged HL7 messages (the necessary tables, access rights, connection).
9) Verify that the JDBC driver for the selected database is available to Mirth Connect.
If Mirth Connect does not have the required JDBC driver (e.g. for PostgreSQL or MS SQL Server), then:
9.1) Place the corresponding .jar file in the folder from which Mirth Connect loads additional drivers (usually the `custom-lib` directory).
9.2) Restart the Mirth Connect service so that the driver is recognized.
10) In the «Destinations» section, create a «Destination Connector» of the «Database Writer» type.
11) Configure the «Database Writer» properties:
11.1) In the «Database Writer» → «Database» section, specify the connection type:
- in the «Driver» field, the driver class (e.g. `com.mysql.jdbc.Driver` or `org.postgresql.Driver`);
- in the «URL» field, the JDBC connection string (for example, `jdbc:mysql://localhost:3306/your_database`);
- in the «Username» and «Password» fields, the credentials for accessing the database.
11.2) When using SSL or other connection parameters, add the corresponding settings in the URL or configuration file.
12) In the «Database Writer» → «Statement» section, specify the command to write data.
To store the entire HL7 message in the message_raw field, it is possible to use the variable `${message.encodedData}` or a similar one.
Mirth Connect automatically substitutes the text of the HL7 message into `${message.encodedData}`.
13) Save and deploy the channel.
Verify that the status is «Deployed» with no errors.
14) Test saving to the database:
14.1) Send a test HL7 message from Epic EHR to the Channel.
14.2) Verify in «Dashboard» → «Messages» that the operation was successful.
14.3) Verify that a new record has  appeared in the database.
